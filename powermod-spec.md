## Revision note — 2026-07-15 audit

**A full cross-document audit** (this spec, the user guide, the state-machine diagram, and a register map written from them) **checked every decision against the others and against what the parts physically do.** Changes are marked inline in this document's existing decision-history style. It is a large revision; this note is the map.

**Contradictions resolved — four, each unimplementable as written:**
- **Missed-wake vs. past-wake** demanded opposite behaviour from an identical observable state → resolved by evaluating the timestamp *at commit*.
- **The reason-code register was destroyed by the act of booting** — powering on overwrote the power-off reason before any host could read it → split into two registers.
- **Emergency cutoff: delay or no delay** — the diagram and the spec disagreed → routed through the delay, one code path.
- **Minimum off-duration was never actually solved.** The gap said "minimum off-duration"; the register adopted to close it was "delay before power cut" — a different mechanism entirely. Nothing prevented a zero-second power cycle.

**BOM corrections — the design described behaviour its parts could not deliver:**
- **TP4056 → MCP73871 → *back to TP4056*, behind a discrete PFET OR (reversed later the same day).** The hot-swap claim does require power-path isolation, but **an OR-ing stage provides it just as well as a power-path IC**, and TP4056's disqualifying defects belong to the *topology* it was assumed to sit in, not to the part. The reversal halves the power stage (**~$2.00 → ~$1.00**) and roughly doubles mains output (1.3A → ~2.7A), because MCP73871's `SEL`-high limit is **1.8A total, "system power and charge current combined"** — a cap we were paying $1.46 to impose on ourselves. **BQ24074 remains disqualified**, and the MCP73871 analysis is retained as history. See "Charger and power path — REVERSED".
- **PCF85063A → RV-3028-C7 → *BM8563*, a PCF8563-class part (reversed 2026-07-15)** — the RV-3028's stated reason was false (no TCXO), and the audit later showed its 40nA never mattered either; what remained was ±1ppm for $1.62/board. Gates run against the clone's own datasheet; wake API unchanged (minute alarm + countdown residue); `BACKUP_TRICKLE` register replaced by a solder jumper. See RTC part decision.
- **14-pin MCU → ATtiny1617 (24-pin VQFN 4×4).** The pin budget missed by four, then grew twice more the same day (charger reversal; the OR's latch fix) — **18 pins, and a 20-pin part would ship with zero spare.**
- **Output 3A → ~1.2A → *a per-mode table* (same day).** 3A was never achievable on a 15W input, and drew double the battery connector's rating; the ~1.2A single figure then died with the charger reversal. **There is no one number: ~2.7A on mains, ~1.0–1.5A on battery via the connector, ~1.8–2.6A via the pads, ~2A in 3.3V mode** — see the ratings table.
- **TPS63020 selected**, and its load disconnect turns out to be *how host power is switched at all*.
- **The OR stage as first drawn latched permanently on unplug** — Q1's VBUS-tied gate held itself on through the battery, running the full load through Q2's body diode. Fixed: Q1 is MCU-driven off a VBUS-vs-VBAT comparison (Q2 stays fully passive), which costs a GPIO and makes the **ATtiny's own WDT a hard requirement**. FETs selected: **2× AON7407** (Q1/Q2), **BSS138** (Q3 driver).

**Added:** opt-in watchdog (the only unrecoverable failure mode), minimum off-duration, critical-battery threshold, LED power specification, a 3.3V logic rail, shutdown-pending status, wake-time read-back, hold-to-force on the button.

**Withdrawn after adoption** — recorded because the reversals are the useful part: **temperature-qualified charging** (the 2-pin connector gives the sensor no path to the cell), and **input-voltage telemetry** (mains detect was a digital read under the then-selected charger, so it wasn't free — **since half-reopened**: the charger reversal made mains detect a VBUS *ADC* divider, so the measurement is free again; deliberately not adopted in protocol v1. See Telemetry scope).

**Narrowed:** host support is **3.3V I2C**, not "any I2C" — a 5V host damages the board.

**Every part claim is now cited or marked unverified.** See the claims register appendix — **45 claims across four batches, 18 were false — and as of 2026-07-15, zero remain open**, and the failure rate *rose* as the claims got more consequential (batch 3: 16 of 23). The earliest two failures had already changed a decision before anyone looked. *(This sentence itself previously read "16 claims checked, two were false" — stale by 21 claims and 14 failures, in the paragraph describing the discipline. ## What PowerMOD is

A generic, host-agnostic scheduled power controller and UPS: real time clock, battery charging and management, and a two-command power protocol (schedule a wake, request a cut) over I2C. Designed to be dropped into any project needing scheduled power-cycling, battery backup, or both — a Linux SBC, a bare-metal MCU project, or anything else with a **3.3V** I2C bus and a power input to control (the 3.3V is a hard limit, not a preference — see Interface).

The design deliberately makes zero assumptions about what it's powering or commanding: no host OS awareness, no GPIO signaling, no protocol behavior that only makes sense for one kind of host. The protocol, the power stage, and the physical connectors were all chosen specifically to keep this true (see Interface and Extensibility sections). **Linux SBCs (Olimex A64-OLinuXino, Raspberry Pi-class boards) are the first documented, validated use case and the primary motivation for building this at all — not a constraint on what the board can do.**

## The problem (as first encountered, via the Linux SBC case)

Existing scheduled-power boards for SBCs (Witty Pi and peers) infer host state instead of letting the host declare it — watching a serial pin voltage, assuming Raspberry Pi OS + Python + specific shutdown hooks, silently overwriting timezone assumptions. This breaks in real, documented ways, even on their officially-supported OS, and breaks harder on anything else (hit first-hand running Alpine Linux, requiring a custom driver). This specific, real pain point is what motivated PowerMOD's core design principle — but the principle itself (below) turned out to generalize far beyond the case that surfaced it.

## Design

The host owns the "when is it safe to cut power" decision entirely. The board never tries to detect or infer host state — it only responds to explicit commands.

**Two core operations:**
1. **Wake me at time T** — host writes a future timestamp before beginning its own shutdown sequence.
2. **Cut power now** — host issues this only once its own shutdown is genuinely complete, however its own OS determines that.

**Board exposes:** RTC (read/write), wake-time schedule, cut-now command, battery voltage, mains-vs-battery status.
**Board never needs to know:** host OS, init system, or how the host determines it's done.

This beats both Witty Pi's approach (fragile signal-inference) and a simpler "cut at time T unconditionally" alternative considered (host-decided cutoff removes the timing-race risk of a board-guessed deadline).

**Fallback safety (host hangs, never says "cut now"):**
- Board exposes mains/battery status so the host can prioritize its own shutdown urgency (low stakes on mains, real urgency on battery).
- Independent hard voltage floor, enforced by the board regardless of host cooperation: below a fixed threshold, cut unconditionally. Not a timeout guess — a physical safety limit, same category as any LiPo protection IC.
  - **"Hard floor" vs. "default disabled" — an ambiguity the register map forced to a decision, now resolved.** This bullet describes the floor as hard, fixed, and enforced regardless of cooperation. The Witty Pi precedent adopted below (register #19) is *default disabled*. Those cannot both be true.
    - **Resolved: the threshold ships enabled at 3.0V** (register map §7, §11). It remains configurable, and disabling it (value `0`) stays legal for an integrator with their own protection circuitry — but opting out of over-discharge protection is an explicit act rather than the out-of-box state.
    - **What settled it: three other resolutions in this document are load-bearing on the floor being active.** The hung-host-on-battery case is bounded by it ("the host either eventually drains the battery, triggering the independent low-voltage cutoff"); the stuck-I2C-bus case concludes "no new design needed" explicitly because of it; and the decision not to extend the watchdog to the battery case rests on it. **Shipping the floor disabled would silently falsify all three for a default board** — each argument would be leaning on a safety net that isn't there unless someone read a register table and switched it on. The "hard floor" language is the load-bearing one; the disabled default was inherited from precedent without noticing it contradicted three conclusions already drawn here.
    - **Why PowerMOD can diverge where Witty Pi could not.** Their likely reason for a disabled default is that their boards are often run with no battery, where an enabled threshold reading ~0V would fire instantly. PowerMOD gates both thresholds on the charger's battery-detect — mains-only mode has no voltage monitoring at all (Section on Scope) — so that failure mode cannot occur here. The divergence is enabled by a design decision they don't have, not a disagreement about risk.
- **Re-arm/hysteresis:** cutoff floor and re-enable threshold are separate values, not one. Board does not restore power the moment voltage creeps back above the cutoff floor — that risks a boot-current draw immediately sagging voltage back under the floor, causing a rapid cut/recover/cut oscillation right when the battery is most fragile. Power is restored only once battery voltage recovers to a meaningfully higher threshold (3.6V) **or mains power is supplied** — the latter giving immediate recovery the moment someone physically intervenes on a dead-battery device, without waiting on trickle-charge.
- **Hung host on mains — reopened, reversed, now an opt-in watchdog (audit finding, adopted).** Previously an explicit non-goal, twice reconfirmed: PowerMOD would not detect, diagnose, or act on host-level hangs at all, and detection was left entirely to the operator's own monitoring (the expected signal being the absence of whatever telemetry the host normally produces during its wake cycles). **That reasoning conflated two different things, and the conflation is what made it look principled.** The core principle (Section 2) forbids PowerMOD *inferring* host state — Witty Pi's actual sin was inferring liveness from a serial pin voltage. A watchdog where the host **explicitly writes to a register to declare itself alive** is not inference; it is the same architecture as "cut now" — an explicit host declaration that PowerMOD merely acts on. It is no more a violation of Section 2 than the two core commands are.
  - **The asymmetry was already unjustified.** The max-sleep-before-forced-wake safety net (Section on explicit scope decisions) is structurally the identical mechanism — a timer, disabled by default, that recovers a host which failed to declare something — applied to the OFF state. There was never a principled reason to accept it there and refuse it in the ON state.
  - **What it closes:** the only unrecoverable failure mode in the entire design. The old resolution's own words concede it — operator telemetry going quiet only *detects* the hang. On the remote, unattended deployments this board is explicitly built for, detection means a site visit; a watchdog means the device fixes itself. On battery the case was already bounded (the low-voltage cutoff eventually force-cycles it); on mains it was bounded by nothing at all.
  - **Requirement: a configurable watchdog timeout register, disabled by default.** If enabled, the host must write to a designated register within the timeout to declare liveness; if the timeout expires, PowerMOD power-cycles the host and records a distinct watchdog-expiry reason code. Zero BOM cost (one timer, one register), and disabled-by-default matches the same disable-by-value convention already used for the low-voltage, recovery, and max-sleep thresholds — so it never surprises an integrator who didn't ask for it.
  - **The watchdog arms on the host's first kick, not at power-on (register-map finding, load-bearing).** If it started counting the moment the rail came up, any host whose boot takes longer than the timeout would be power-cycled before it ever finished booting — permanently. A 60s timeout on a 90s-booting host is an unbreakable loop that bricks the device in the field, and unreachable over I2C precisely because the host never gets far enough to talk. **Rule: after every power-on the watchdog is disarmed; it arms on the first kick and stays armed until power is cut.** A host that never kicks is never watchdogged, which is also the correct behavior for host software that doesn't know the watchdog exists. Costs one flag, and removes any need for a separate boot-grace-period register.
  - **Still a non-goal, and this is the line:** PowerMOD does not *diagnose* anything. It does not know or care *why* the host stopped writing — crashed, hung, deadlocked, or deliberately silent are indistinguishable to it and treated identically. Operator-side telemetry monitoring remains the right tool for understanding *what* went wrong; the watchdog only restores service without waiting for a human.

**RTC backup power — optional and user-fitted, justification narrowed (audit revision).** PowerMOD provides two labelled solder pads wired to the RTC's backup supply — since the 2026-07-15 RTC reversal, via diode D4 into the BM8563's diode-OR'd VDD (formerly the RV-3028's dedicated VBACKUP pin) — not to an ad-hoc backup rail. **The board does not ship a backup source; the integrator fits one, or doesn't** (see "Backup fitting" below for why, and what to fit).

**Justification, corrected.** The earlier reasoning claimed the cell was "load-bearing for mains-only mode, redundant-but-harmless for LiPo mode." That over-claimed in one direction and under-claimed in the other, and it rested on a storage error corrected below.
- **What only the coin cell can do: keep the RTC counting through a total loss of power.** Nothing else on the board can do this — it is irreducibly a battery problem. Everything else the cell was previously credited with backing (pending wake time, power-on state bit) belongs in EEPROM and is corrected below.
- **Where that actually matters, stated honestly:** hosts that use wall-clock scheduling *and* cannot re-sync time on boot (no network, no GPS). A networked host re-syncs via NTP and never notices; a host using relative-time scheduling (Section on RTC default state) does not care what the clock reads at all. This is a real set of deployments, but a narrower one than "all mains-only deployments."
- **Off-mains LiPo swap — the case that decides it.** On mains, replacing the LiPo needs no coin cell whatsoever: **the PFET OR** (Section on charger and power path — REVERSED; *was* the power-path IC) keeps the board and RTC alive from the mains rail while the battery is genuinely isolated. **The charger reversal does not weaken this — it strengthens it; see the hot-removal claim below.** **Off mains, pulling the LiPo is a total power loss, and the coin cell is the only thing that preserves the clock across it.** A technician replacing a degraded cell at a remote solar site with no mains present is a realistic servicing scenario for the flagship deployment, and it is what keeps the cell in the design.
- **Interaction with the power-on default (worth stating, since it changed):** adopting Restore-previous-state as the default *raises* the cost of losing the RTC. Under the old Always-ON default, any total power loss recovered by booting regardless. Under Restore, a total loss during a scheduled sleep leaves the device off — so preserving the clock and schedule matters more than it did when the cell's justification was originally written.
- **Storage correction (this was simply wrong before):** the pending wake time and the power-on state bit are **EEPROM state, not coin-cell-backed state.** They are non-volatile and survive a dead coin cell for free. Only the RTC's *count* depends on the cell.
  - **Write-endurance consequence — missed on the first pass, now resolved.** The correction above originally claimed these "reuse the wear-leveling infrastructure already built for configuration and the event log." **That was asserted, not traced, and it is false**: configuration is not wear-levelled (it is written rarely), and the event log's wear-levelling is specific to its own 8×5-byte slot arithmetic. Neither covers a separate state bit or wake time. **Both are written on every power cycle** — the state bit twice (on and off), the wake time once — against roughly 100k-cycle EEPROM endurance:

    | Duty cycle | State bit life | Wake time life |
    |---|---|---|
    | Every 15 min | **1.4 years** | 2.9 years |
    | Every 30 min | 2.9 years | 5.7 years |
    | Hourly | 5.7 years | 11.4 years |
    | 4× per day | 34 years | 68 years |

    **Anything more frequent than roughly hourly fails inside its service life**, and Restore-previous-state — the mode that needs the state bit — is now the *default*. Note the irony, because it is the point: **the original coin-cell-backed-RAM design had no wear problem at all.** Moving this state to EEPROM was correct for surviving a dead cell and traded one failure mode for another without anyone noticing.
  - **Resolved, and the state bit costs nothing to fix: derive it from the event log instead of storing it.** The log already records every power transition, is already wear-levelled, and its reason enum is already self-describing (bit 7 = power-on vs. power-off — register map §8.2). So "was the host powered when power was lost?" is answered by scanning back to the most recent non-informational entry and reading one bit. **Zero additional writes, no new state, and it cannot disagree with the log.** Empty log (first ever power-up) or a log holding only informational entries → default to ON, matching the specified first-power-up behavior.
    - **⚠️ Corrected by the spec walk (2026-07-15): the naive derivation contradicts semantic-review finding 1.** "Most recent event is a power-off ⇒ host was off" reads `LOW_VOLTAGE_CUTOFF` as *off* — but finding 1 requires an emergency cutoff to record the host as **ON** (cut involuntarily). **The derivation classifies by voluntariness, not direction:** a *deliberate* cut (`HOST_REQUEST`, `BUTTON`, `BUTTON_FORCED`) derives OFF; an *involuntary* cut (`LOW_VOLTAGE_CUTOFF`, `WATCHDOG_EXPIRY`) derives **ON**; any power-on event derives ON. The watchdog case composes correctly for free: if total power dies between a watchdog cut and its automatic recovery, the derivation boots the host on power return — which is exactly what the watchdog was trying to do.
  - **Resolved for the wake time: wear-level it across a small rotation of slots**, exactly as the event log does — the same standard technique, applied to the storage that actually needed it rather than assumed to cover it. Sizing the rotation is a register-map/firmware detail; the requirement is that a per-cycle write must not land on the same byte every time.
- **Coin cell vs. supercap — settled 2026-07-15 in favour of the coin cell, on measured numbers — and the conclusion survives the RTC reversal unchanged** (the backup draw rose 45nA → 250nA, but the decision always rested on *supercap leakage and coin self-discharge dominating either figure by 10–100×*, which is exactly why the cheaper RTC was safe to adopt). Original analysis: RV-3028-C7's **40nA** backup current and dedicated VBACKUP pin (both datasheet-verified, along with its automatic switchover and trickle-charger function) made a supercap genuinely viable — no holder (one of the pricier mechanical parts), no shelf life, no user-replaceable component with its own documentation burden and failure mode. The catch is supercap self-leakage. **Now measured against a real datasheet rather than estimated** (Abracon ADCH-S05R5S coin supercaps; leakage max, 72h at rated voltage), charging to 3.3V and discharging to the RV-3028's 1.2V minimum:

| Supercap | Leakage (max) | RTC's share of drain | Backup |
|---|---|---|---|
| 0.1 F | 3 µA | 1.3% | ~19 hours |
| 0.22 F | 4 µA | 1.0% | ~32 hours |
| 0.47 F | 6 µA | 0.7% | ~45 hours |
| 1.0 F | 8 µA | 0.5% | ~73 hours |
| 1.5 F | 12 µA | 0.3% | ~73 hours |

- **The RTC is 0.3–1.3% of the drain.** The 40nA figure that makes the supercap sound viable is almost irrelevant — the capacitor's own leakage is 75–300× larger and sets the answer entirely.
- **Backup plateaus at ~3 days, and buying a bigger part does not help.** Leakage scales with capacitance, so 1.5F backs up no longer than 1F. This is the part the earlier "days, not years" hedge missed: it isn't a knob that can be turned up. **If more than ~3 days is ever needed, the supercap route is closed outright** — no amount of capacitance reaches a week.
- **A CR2032 at 40nA is limited by shelf life (~10 years), not by the load** — the RTC would take centuries to drain it. The two options are not on the same scale and no sizing exercise brings them together.

**Where this lands: the supercap covers a battery swap (minutes) and an ordinary grid outage (hours) comfortably. It does not cover a week of winter darkness at a solar site, and cannot be made to.** So the question was never "how big a supercap" but "is a multi-day ceiling acceptable" — a question about deployments, not components.

### Backup fitting: two labelled pads, no holder — board area was never costed, and it dwarfs everything (adopted 2026-07-15)

**The holder's *price* was weighed twice in this document. Its *area* was never mentioned once — and area is by far its larger cost.**

| | Board area |
|---|---|
| ~~ATtiny1616~~ + RV-3028-C7 + TPS63020 + ~~MCP73871~~ — every IC, **as costed before the charger reversal and the 1617** | **~42 mm²** |
| Same, **after the reversal + the 1617** — TP4056 (ESOP-8 ~19mm²) + 2× AON7407 (DFN3×3, 9mm² ea) + BSS138 (SOT-23) replace one 4×4 QFN; MCU 3×3 → **4×4** (+7mm²) | **~74 mm²** (**all parts now selected — measured, not estimated**) |
| **CR2032 cell alone** (before the holder's retention arms and solder tabs) | **~314 mm²** |

> **The reversal roughly doubles the IC area, and that is a real cost this document should not hide.** MCP73871's 4×4 QFN was *integration you could measure*: one 16 mm² part became a ~19 mm² ESOP-8 plus a discrete OR stage. **If the PFETs must grow to handle Q1 at 2.7A (they likely must — see the FET sizing table), this gets worse, not better.** The ~$1.20 saved per board is partly paid back in silicon area, on a board whose defining constraint is size.
>
> **Corrected twice on 2026-07-15, in both directions: ~85 mm² (guessed) → ~67 mm² (PFETs chosen) → ~74 mm² (1617's 4×4 replaces the 1616's 3×3).** That guess assumed SOP-8 PFETs; the selected AON7407 is DFN-8 3×3 (9mm²), materially smaller. **The error was pessimistic rather than optimistic, which is the safer direction to be wrong — but it was still a guess presented in a table of measurements.**
>
> **The conclusion survives, and stays a footnote:** 314 mm² still dwarfs 74 mm². The cell drops from ~7.5× to **~4.2×** the combined IC footprint — less dramatic, still decisive. **A weakened argument that reaches the same conclusion must be re-stated at its true strength, not left quoting the old multiplier.**

**The cell is ~3.2× the combined footprint of every integrated circuit in the design** (re-computed after the RTC reversal: the BM8563+crystal+diodes add ~25mm² over the ceramic RV-3028, IC total ~74→~99mm²) **and ~16× the area of the RTC subsystem it exists to back up.** Each recomputation has weakened this ratio — 7.5× → 4.2× → 3.2× — and it still clears decisively; the day it approaches 1× the pads decision reopens. The holder ships on **44mm** tape and reel (Keystone's catalogue, 20mm-cell SMT holders); every IC here ships on 8–12mm. **It is, comfortably, the largest component on the board** — and a design that has agonised over a 3×3 versus 3×4mm package elsewhere never noticed.

**Resolved: replace the holder with two labelled solder pads (`VBACKUP`, `GND`), wired to the RTC's backup diode D4 (the diode-OR replaced the RV-3028's dedicated VBACKUP pin in the 2026-07-15 part reversal) — formerly the RV-3028's VBACKUP pin.** The backup becomes user-fitted and genuinely optional — which matches what it always was: irrelevant to relative-time hosts, irrelevant to networked hosts, load-bearing only for non-networked wall-clock deployments. **This reclaims the single biggest area consumer on the board and deletes a mechanical part, a user-replaceable item, and its documentation burden, in exchange for nothing that most integrators use.**

- **The pads accept whatever the deployment needs:** a lead-terminated or solder-tab coin cell (years of backup), a supercap (~1–3 days, and now much more attractive — it was only ever rejected on that ceiling, and the pads remove its other disadvantage by removing the holder from the comparison), or **nothing at all**, which is the cheapest and correct answer for most.
- **⚠️ The documentation must forbid soldering a bare coin cell, explicitly.** Heating a lithium coin cell can vent or rupture it. Pads invite exactly that mistake from anyone who owns a CR2032 and an iron. **Only solder-tab or lead-terminated cells, or a supercap.** This is a safety instruction, not a preference, and it is the one real cost of this change.
- **The pads need polarity marking and, ideally, asymmetric geometry** — otherwise they inherit the battery connector's reversal problem with no key at all. Unlike the JST-PH case, silkscreen genuinely *is* sufficient here: a bare pad is unambiguous precisely because nothing pre-wires it for you.
- **Interaction with the temperature range (Section 8):** the coin cell's −30/+60°C was the narrowest limit on the board. **Making it optional widens PowerMOD's own range back to the silicon's −40/+85°C** for anyone who fits a supercap or nothing — an unplanned second benefit.

**"Coin cell vs. supercap" stopped being a board decision the moment the pads were adopted — and that reframing is the real resolution (2026-07-15).** This section previously ran a long argument to pick *one* backup for the board to ship, concluded "keep the coin cell," and justified it as "paying for a holder to cover the deployment this board exists for." **With no holder on the board, that conclusion answers a question that no longer exists.** The pads decision was inserted above it without rewriting the argument it invalidated — the same failure this document keeps recording elsewhere, committed here.

**What the analysis above is actually for, restated:** it is no longer a choice PowerMOD makes, it is **guidance for the integrator making it**, and every number in it still earns its place:

| Fit | Backup | When it's right |
|---|---|---|
| **Nothing** | none | Relative-time scheduling, or any host that re-syncs its clock. **Most deployments** — and the cheapest, smallest answer. |
| **Supercap** | ~1–3 days (hard ceiling) | Grid outages and off-mains LiPo swaps. One part, no shelf life, nothing user-replaceable, and the RTC's own trickle charger drives it with no support circuitry. |
| **Coin cell** (solder-tab or lead-terminated) | years | Non-networked wall-clock hosts; **the flagship remote-solar case, which can be dark for a week — precisely where the supercap's ceiling silently isn't there.** |

**So both of the earlier conclusions survive, as advice rather than as a board-level commitment:** the supercap really is the better engineering choice for most deployments, and the coin cell really is the only thing that covers a week of winter darkness. **The pads let both be true at once, which is why the decision they replaced was the wrong shape** — it forced one answer onto a board serving deployments with genuinely different needs.

**Worth recording for anyone revisiting this:** if PowerMOD ever ships a variant that is explicitly *not* for long-dark deployments — a bench/indoor scheduler, or a mains-only-only product — **the supercap becomes the better answer and the numbers above already justify it.** Nothing about that decision needs re-deriving; only the deployment assumption changes.
- **Scope of what the coin cell backs**: the RTC count only. **Does not** power the LEDs, and **does not** enable PowerMOD to actually power the host — a wake time surviving a total outage does not mean the host will wake on schedule if no real power source (mains or LiPo) is available when that time arrives. This distinction should be stated clearly in user-facing documentation to avoid the coin cell being oversold as "outage-proof" beyond timekeeping.
- **Missed wake time during a total outage — resolved, then found to contradict the past-wake rule, now re-resolved (audit finding).** Concretely: a wake time is scheduled for 2 PM; both mains and battery are unavailable from 1 PM to 4 PM (coin cell keeps the RTC alive, but there's no power to actually switch); power returns at 4 PM. The original resolution was: **wake the host immediately** rather than discarding the missed alarm, consistent with the unattended-recovery philosophy running through the rest of this design.
  - **The contradiction:** the Section on invalid/past schedule writes independently resolved that **a past wake time is simply never treated as due**, explicitly *rejecting* "wake immediately" — because a host that writes a bad timestamp would otherwise get an instant, unexplained reboot. Both rules are individually well-argued and they cannot both be implemented, because **at wake time the two cases are physically indistinguishable**: PowerMOD powers up, reads a stored timestamp that is in the past, and has no way to know whether that's because power was out during the window or because the host wrote garbage. Same observable state, two opposite required behaviors.
  - **Re-resolved: evaluate the timestamp when it is committed, not when it comes due.** This satisfies both rules' actual intent at the cost of one bit, and it reuses the shadow-register-and-commit path already required for write atomicity (Section on state-machine details):
    - **Already in the past at commit time → mark expired. Never fires, ever.** This is the bad-timestamp case, and it preserves the past-wake rule's intent: a miscalculated write causes the host to stay asleep, not to reboot mysteriously. The max-sleep safety net remains the backstop.
    - **In the future at commit time → arm it.** If it then goes by while PowerMOD has no power to act, it is a genuinely missed alarm and **fires as soon as usable power returns.** This preserves the unattended-recovery intent: the host asked for something reachable, and physical reality is the only reason it didn't happen.
  - The distinction the original two rules were each reaching for is exactly "was this reachable when the host asked for it" — which is knowable at commit time and unknowable later. Cost: one flag stored alongside the wake time in EEPROM.
- **LiPo hot-removal — claim was false against the original BOM, now made true by the power-path IC (audit finding).** The claim: when mains is connected, the battery is disconnected from the power rails and only charges — the host's power is never sourced from the battery while mains is present — so removing the LiPo on mains is safe and does not interrupt host power.
  - **This was not true of the TP4056-class charger originally speced.** That part has no separate system rail; the load hangs permanently off the BAT pin, in parallel with the battery. The battery was never isolated, and hot-removal would have dropped the host onto a current-limited charger output and browned it out. The architecture described here was always *power-path management*, which the speced charger does not do — see the Section on power stage for the full finding and the resolution.
  - ~~**With the power-path IC now adopted, the claim is accurate as written**~~ — **this was still wrong, and the charger reversal is what finally made it true (2026-07-15).**
    - **MCP73871 did *not* genuinely isolate the battery.** Its own datasheet: *"if the system demands more current than the input can provide, the ideal diode becomes forward-biased and **the battery may supplement** the input current to the system load."* So under peak load the cell was in the path *while on mains* — the exact thing the claim denies. **The audit adopted a part to make a safety claim true, and the part didn't make it true.** It was only ever true below the input current limit, which was never stated.
    - **The PFET OR does isolate it, structurally.** With VBUS at 5V and the cell at ≤4.2V, Q2's body diode is reverse-biased and its gate sits at VBUS: the battery is **off the rail, not merely deprioritised on it**. Hot-removal on mains is safe because the cell was never connected, not because something arbitrated in its favour.
    - **One honest exception, which must be documented rather than buried:** if the supply is undersized and VBUS sags *below* the cell voltage, Q2 conducts and the battery carries the load. **"Isolated on mains" holds for an adequate supply; with a weak one the cell is in the path** — see the USB-C input section.
    - **Three passes to get one safety claim right.** It was false against TP4056-alone (no isolation at all), false against MCP73871 (isolation only below the current limit), and true against TP4056-behind-an-OR. **Each pass sounded conclusive.** Anything user-facing should state the exception, not the headline.

## Scope

**Design scope: any host with 3.3V I2C — an honest, bounded claim, not "anything" (external review feedback adopted; boundary narrowed 2026-07-15, see below).** PowerMOD's protocol, power stage (now buck-boost with selectable 3.3V/5V output — see Extensibility section), and physical interface were all chosen to avoid host-specific assumptions, within that I2C boundary. This isn't aspirational — it's the actual, current design, not a future goal. **Worth being explicit about the real limit this implies**: I2C covers essentially all MCU families (RP2040, ESP32, STM32, AVR) and Linux SBCs, but does *not* cover hosts that don't expose I2C conveniently or at all — routers, many mini PCs/NUCs, and other USB-or-Ethernet-only equipment.

- **Narrowed 2026-07-15: the boundary is 3.3V I2C, not I2C.** PowerMOD is a **3.3V** I2C device (see Interface), so **a 5V host does not simply fail — it damages PowerMOD**, and needs an external level shifter. In practice this costs almost nothing against the stated targets: Linux SBCs (including the flagship Pi and A64), RP2040, ESP32 and STM32 are all 3.3V, as is the STEMMA-QT ecosystem the connector belongs to. **The casualty is the "AVR" claim above** — classic 5V AVRs like an Arduino Uno are exactly the case that breaks, and this document listed them as supported. They remain usable *with a level shifter*; they are no longer plug-and-play, and saying "any host with I2C" without that qualifier was over-claiming. "Generic" in this spec means "host-agnostic within the I2C-capable world," not literally universal — a genuinely different transport (USB serial, a simple GPIO command interface) would be needed to reach that further tier of hosts, and is explicitly out of scope for this design (same protocol logic could in principle be re-exposed over a different transport later, but that's a separate, unbuilt variant, not implied by anything here).

**Flagship documented use case: Linux SBC hosts** (Olimex A64-OLinuXino, Raspberry Pi-class boards). This is where PowerMOD's real UPS/charging behavior, mains/battery awareness, and integration guidance are most thoroughly documented (Sections on Interface, host integration patterns), because it's the case that motivated the project and the one with the deepest validation (Section 8's evidence log). It is a *use case*, not a *constraint* — the underlying design does not assume Linux, an SBC form factor, or any particular host capability.

**Bare-metal/MCU hosts — fully supported by the protocol and power stage, lighter documentation.** The two-command protocol (wake/cut, plain I2C) needs zero changes to work with a bare-metal host, and — per the analysis that follows — integration is often *simpler* than the Linux case, since a bare-metal application already knows unambiguously when it's safe to lose power, with no OS shutdown sequence to reason about. A real, independently-built, awarded project (Hackaday's 2026 Green Powered Challenge "powerTimer") validates this exact two-command pattern on a bare-metal MCU host. **What genuinely differs by host class is the power stage's operating point, not its topology or output flexibility** — a battery-and-charging-heavy UPS is naturally sized for SBC-class current draw (~1.2A, revised down from 3A — see the power stage section) and won't hit the very lowest quiescent-current numbers a pure microwatt-class MCU application might want (see table). This is a real, physical tradeoff (the same components that enable real UPS/charging behavior have their own baseline overhead), not a documentation gap — a future, separate ultra-low-power variant remains a valid idea if that specific tier is ever wanted, but it doesn't block using PowerMOD as-is for the vast majority of MCU projects today.

|---|---|---|
| RTC | **BM8563** (PCF8563-class + crystal, ~±20ppm — **not** temp-compensated; reversed from RV-3028-C7 2026-07-15) | Same class — *the RTC stopped differentiating the variants once the audit showed backup life is leakage-dominated, see below* |

**Table revised (audit).** Two rows changed meaning:
- **The RTC row no longer differentiates anything.** RV-3028-C7 was listed here as the hypothetical future variant's part; it has since been adopted for the main design (see the RTC decision below), so both columns are the same part. The row is retained only to make that explicit rather than leaving a stale contrast implied.
- **The quiescent-budget argument now rests entirely on the buck-boost and the charger**, which is where it always actually lived. The RTC was never the reason PowerMOD can't reach nanoamp-class quiescent current — a 40nA RTC is compatible with either design. The real, unavoidable contributors are the buck-boost converter and the power-path charger IC, whose own baseline overhead is the price of real UPS behavior.

These still aren't tunable values on one circuit — the buck-boost converter and charger IC that make the SBC case work are exactly what prevent nanoamp-class quiescent current, so it's a different topology, not a config option. The point of the table is unchanged; only the honest attribution of *which* parts drive it has been corrected.

**RTC part — REVERSED 2026-07-15: RV-3028-C7 → BM8563 (PCF8563-class) behind a silicon diode-OR.** The section below is retained as history; its analysis of why the RV-3028 was *originally* chosen already documents that the stated reason (a TCXO) was false. The reversal closes the loop:

- **What the $1.73 was actually buying, after the audit stripped the false claims: ±1ppm factory calibration and on-die integration. Nothing else survived scrutiny** — the celebrated 40nA never mattered (supercap leakage and coin self-discharge dominate backup life by 10–100×), the TCXO never existed, and the UNIX counter is ~30 lines of BCD in firmware.
- **Gates run against the BM8563's own datasheet (GATEMODE edition), all passed:** PCF8563-identical register map including the countdown timer; alarm at minute granularity (firmware arms the minute alarm and burns the 0–59s residue on the 1Hz countdown timer — **the wake-to-the-second API is unchanged**); **SCL/SDA and INT rated −0.5 to +6.5V *independent of VDD***, which is what makes the diode-OR legal; backup draw **250nA typ @3V** (noise, per the leakage analysis); clock-integrity floor 1.2V (diode-OR'd coin cell delivers 2.3V).
- **Backup switchover is now diode physics, not a register** — 2× 1N4148-class silicon (nA leakage; the 0.7V drop is affordable on both legs since VIH = 0.7×VDD tracks the OR'd rail). **The "decorative pads" failure class from the electrical walk dies with it**: nothing needs enabling.
- **`BACKUP_TRICKLE` (`0x51`) is removed from the register map after a lifetime of hours** — supercap charging is a solder-jumpered resistor from 3V3, which is *more* coin-cell-proof than any register: no write can ever charge a primary cell.
- **What is genuinely given up:** ±1ppm → **±20ppm-class initial tolerance** (~30 s/year → ~1 min/month at 25°C; the cold parabola adds as it always did — guide §1.2 rewritten); **the deferred temperature-compensation option dies** (no offset register to trim — the ATtiny's die sensor stays free but its correction target is gone); ~+25mm² of board (TSSOP-8 + crystal + diodes vs. one ceramic).
- **New delivery-default trap, third of its class: CLKOUT ships enabled at 32kHz** (open-drain, burning backup-scale current and radiating). Firmware disables it unconditionally at init.
- **Vendor identity — resolved same day:** both vendors' datasheets are now read and gate-identical (claims row 44). **Primary: Shanghai Belling (C194063, $0.056)** — the cheaper part, with M5Stack's entire product line as field evidence; GATEMODE (C269877, $0.108) is the equally-verified alternate.
- **Net: −$1.67/board, the sole-source part becomes commodity multi-source silicon with two gate-verified vendors, and the BOM falls ≈ $4.50 → ≈ $2.90.**

*(Historical section follows.)*

**RTC part — changed to RV-3028-C7, but the reason given for changing it was false (datasheet-verified 2026-07-15).** The user guide advertised a "factory-calibrated, temperature-compensated realtime clock," while the PCF85063A originally speced is a plain crystal RTC with a single-point offset register. The claim and the part contradicted each other — that much was correct. **The audit then resolved this "by changing the part rather than retracting the claim," asserting that RV-3028-C7 "has an integrated TCXO (~±1ppm)." It does not. That assertion was wrong, and the part change did not fix the problem it was made to fix.**

- **What the datasheet actually says.** "Time accuracy: Factory calibrated to **±1 ppm @ 25°C**" — a calibration figure at a *single temperature*, not compensation across temperature. §4.15 is explicit that the offset correction "has only the effect of shifting the time vs. temperature curve vertically up or down. **It has no effect on the time vs. temperature characteristics of the final frequency.**" The oscillator parameters give the curve: **−0.035 ppm/°C² × (T−25)²**, turnover 25 ±5°C — a textbook *uncompensated* 32.768 kHz tuning-fork parabola.
- **So the swap changed nothing on this axis**, because both parts use the same class of crystal and the parabola is a property of the crystal cut, not the chip. Real compensation requires a temperature sensor and active correction, which this part has no more than PCF85063A did. Resulting drift: ~20 s/month at 10°C or 40°C, **~57 s/month at 0°C, ~111 s/month at −10°C**.
- **Resolved the way it should have been the first time: retract the claim.** The user guide no longer says "temperature-compensated"; it states the ±1 ppm @ 25°C figure, publishes the drift table, and explains that relative-time scheduling is immune to it and a re-syncing host doesn't care — which is most integrators. **This was one of the options considered and passed over on the strength of the false TCXO claim.**
- **Where the claim came from, and the competitive problem it exposes (found while verifying an unrelated claim, 2026-07-15).** Witty Pi 4 L3V7's manual leads with: "**Factory calibrated and temperature compensated realtime clock with ±2ppm accuracy**." The user guide's original wording — "factory-calibrated, temperature-compensated realtime clock" — is **very nearly that sentence**. The claim appears to have been carried over from a competitor's feature list, describing a capability they have and PowerMOD never did. That is how it survived review: it read like a specification because it *was* one — someone else's.
  - **And they genuinely have it.** Their manual: "firmware also makes temperature compensation for the crystal," using a **dedicated LM75B sensor with 0.125°C resolution**. This is not marketing; it is a sensor plus a firmware correction loop.
  - **So PowerMOD is now measurably worse than Witty Pi on RTC accuracy over temperature** — their ±2ppm *compensated* against our uncompensated parabola (~22ppm at 0°C, ~43ppm at −10°C). **On the exact axis this product exists for: waking at the right time, outdoors, unattended.** Worth stating plainly, because the rest of this spec is built on being better than Witty Pi, and here we are not.
  - ~~The fix costs nothing in parts and is deliberately deferred, not foreclosed~~ — **FORECLOSED by the RTC reversal (2026-07-15): the BM8563 has no offset register, so the correction target is gone.** The ATtiny's die sensor remains free, and firmware can still compensate *wake times* arithmetically — but the free-running clock is no longer trimmable. Restoring the option means restoring a trimmable RTC (PCF85063A-class, +$0.43), which is now the recorded price of temperature compensation. (Historically: everything needed was on the board — the ATtiny's `TEMPSENSE0`/`TEMPSENSE1` in SIGROW, and RV-3028-C7 exposed a programmable offset (`EEOffset`: ±0.954 ppm steps, ±244 ppm range, non-volatile). Firmware reads temperature, evaluates the known parabola, and writes the offset periodically — **zero BOM, zero pins, no new hardware.** This is exactly Witty Pi's approach; they simply spend ~30¢ on a dedicated LM75B for better sensor accuracy.
  - **What it would actually buy, since the benefit is bounded by sensor accuracy, not by the idea:**

    | At 0°C | Uncompensated | ATtiny internal sensor (free) | LM75B-class (~30¢) |
    |---|---|---|---|
    | Error | 21.9 ppm | ~8.8 ppm (±5°C sensor) | ~3.5 ppm (±2°C sensor) |
    | Drift | 57 s/month | ~23 s/month | ~9 s/month |

  - **Not adopted, deliberately.** The claim is retracted rather than implemented, because the drift is **irrelevant in relative-time scheduling** — the recommended mode, immune by construction — and any host that syncs from a network corrects it faster than it accumulates. The real cost was never the part; it is firmware plus a thermal characterisation that would have to be validated on hardware, spent on a problem the recommended usage doesn't have.
  - **The door is held open at no cost.** Should wall-clock accuracy outdoors ever become a genuine requirement rather than a sentence copied from a competitor, **this is a firmware release, not a board revision** — no BOM change, no pin budget, no schematic work. That is the reason to record the mechanism here in full rather than just noting the gap.
- **Keep RV-3028-C7 anyway — it earns its place on grounds that are real, just not the one it was chosen for:** **40 nA** at 3V (verified; ~5× better than PCF85063A-class, and load-bearing for the coin cell/supercap question); **integrated XTAL** so no external crystal or load-cap matching; **±1 ppm factory calibration** at 25°C with no user trim; a dedicated **VBACKUP pin with automatic switchover and trickle charger** — precisely the coin-cell interface this design needs; and a **native UNIX time counter**, which matches the register map's uint32 epoch format directly rather than requiring BCD conversion in firmware. **If genuine temperature compensation is ever wanted, it is a different part entirely** (a real TCXO RTC) — verify against a datasheet before committing, and note the audit's whole point is that this was not done last time.

**Two supported operating modes, one board:** (1) **LiPo/UPS mode** (default/primary) — full UPS behavior as described throughout this spec (charging, voltage monitoring, emergency cutoff, hysteresis). (2) **Mains-only mode** — no battery connected; PowerMOD functions purely as a scheduled power switch (wake/cut on command or schedule), with none of the UPS-related behavior (no voltage monitoring, no emergency cutoff, no charging) applicable since there's no battery to protect or charge. Both modes share the same board and protocol — mode is simply a function of whether a LiPo is physically connected, not a separate configuration to set. See also: RTC/coin-cell backup below, which applies to both modes.

- **How "is a LiPo connected" is actually determined — was undefined, now resolved (audit finding).** The claim that mode "is simply a function of whether a LiPo is physically connected" quietly assumed a detection mechanism that this spec never defined. It is load-bearing in two places at once: it selects the operating mode, *and* it drives the Battery LED's "Off = no battery present" state (Section 9). **The naive implementation — treat a very low voltage reading as "absent" — cannot distinguish an absent battery from a deeply discharged one**, which are different situations that need different signals to the operator (Off vs. fast-blink-critical). ~~**Resolved: use the power-path IC's own battery-detection function**~~ — **this resolution died with the charger reversal (2026-07-15) and was re-resolved.** It was closed by pointing at MCP73871's `STAT1`/`STAT2` decoding; TP4056 has no such function, and neither does Witty Pi's equivalent design.

**Re-resolved: pull TP4056's `CE` low and run a ~500ms decay-delta test on the VBAT node.** ~~No cell → the 4.7µF bypass collapses through a bleed resistor in milliseconds~~ — **the BOM walk falsified that (2026-07-15): the only bleed path is the 1.33MΩ sense divider, so τ = 4.7µF × 1.33MΩ ≈ 6.2 seconds.** A dedicated fast-bleed resistor would fix the time constant and destroy the ~5µA standby floor, which is the wrong trade. **The test that actually works: sample VBAT, wait ~500ms with `CE` low, sample again — an absent cell's node sags ~300mV (7.6% of a 6.2s decay); a real cell moves under a millivolt.** Still a positive test, still one GPIO; it just takes half a second instead of an imagined millisecond. Datasheet basis: *"Pulling the CE pin to low level will put the TP4056 into disable mode"* (Pin 8).

**This is better than what it replaces, not merely equivalent.** The original objection was that a voltage threshold "cannot distinguish an absent battery from a deeply discharged one." A `CE` test is a **positive measurement of the cell's presence** — a discharged cell still holds the node up, an absent one cannot — so it answers the exact question the threshold approach could not. **Cost: one GPIO and one resistor**, and the same pin doubles as firmware VPCC (see the power-stage section).

**Caveat, stated because it is not derived from any datasheet: this technique is an invention of this audit, not a vendor-sanctioned pattern.** It must be validated on hardware before the LED scheme and mode selection are allowed to depend on it. Everything else in this decision rests on quoted datasheet behaviour; this does not.

## Interface

- **Data/command:** I2C over a small connector (STEMMA-QT-style 4-pin), not a full GPIO header.
- **I2C bus voltage — never specified anywhere, and it is a host-damage hazard (audit finding, 2026-07-15).** The connector was chosen for *mechanical* reasons — small, not a 40-pin header, no host-specific footprint (Section on connector/mounting). **Its electrical implications were never traced.** No document states the bus voltage, the MCU's supply rail, or where the pull-ups go. Three things now need deciding together, and they are coupled:
  - **STEMMA-QT/Qwiic is a 3.3V ecosystem convention**, and its 4th pin carries VCC. Adopting the connector's *shape* while ignoring its *convention* invites someone to plug a 3.3V Qwiic device into a port that may not be at 3.3V.
  - **The MCU's rail is unstated.** ATtiny1617 runs 1.8–5.5V and would naturally sit on SYS, which is **3.0–5.5V depending on cell charge and mains presence** — i.e. a *moving* logic level.
  - **The hazard: I2C is open-drain, so the pull-ups set the bus voltage.** If PowerMOD fits pull-ups to its own SYS rail, the bus idles at up to ~5.5V. **A Raspberry Pi's I2C GPIO is 3.3V and not 5V-tolerant — this destroys the host.** The flagship use case is the failure case.
  - **This is the same failure as the USB-C CC resistors and the MCP73871 `THERM` bias**: a connector decision whose electrical requirements are invisible in a requirements document and fatal on a board — except this one damages the *host*, not PowerMOD.

**Resolved (2026-07-15): PowerMOD is a 3.3V I2C device. No pull-ups fitted; logic section on a dedicated 3.3V LDO.**

1. **PowerMOD fits no I2C pull-ups at all.** The host's own pull-ups define the bus. PowerMOD drives SDA/SCL open-drain only and must never *source* current onto them — which is what an I2C peripheral should do regardless, and it means PowerMOD cannot impose a voltage on someone else's bus.
2. **The logic section runs from a fixed 3.3V LDO, not from SYS.** **This is not optional, and the reason is the sharp part:** ATtiny1617's `VIH = 0.7 × VDD` and its TWI has **no SMBus-style fixed input levels** (its SMBus support is timing only — timeouts, SDA hold, quick command). So a SYS-powered MCU has a logic threshold that *moves with the rail*, and against a 3.3V host bus the breakpoint is **4.71V**:

   | MCU VDD (= SYS) | VIH | 3.3V bus |
   |---|---|---|
   | 3.0–4.2V (on battery) | 2.10–2.94V | reads HIGH ✅ |
   | ~4.7V (on mains) | 3.29V | **on the edge** |
   | 5.0–5.5V (on mains) | 3.50–3.85V | **reads LOW — bus dead** ❌ |

   **The bus would work on battery and fail on mains** — intermittent, environment-dependent, and it would present as a flaky connector for weeks before anyone suspected logic thresholds. At a fixed 3.3V, `VIH` = 2.31V and the bus is solid at every cell voltage.
3. **Two problems this LDO also solves, which is most of why it earns its place:**
   - **The battery ADC needs a stable reference.** With VDD tracking SYS, using VDD as the ADC reference makes every battery reading scale with the rail — the cutoff and recovery thresholds would drift with mains presence. A fixed rail (or the internal bandgap) is required regardless of the I2C question.
     - **↪ PY32F003 (instock-parts branch) — FIRMWARE REQUIREMENT to preserve this.** The ATtiny provided a true internal fixed ADC reference; the PY32's reference is **VDDA (= the 3.3V LDO rail)**, which holds 3.3V until the cell empties and then sags with LDO dropout — so raw readings would drift exactly as warned here, worst at the low-battery cutoff where accuracy matters most. **Firmware must sample the internal 1.2V bandgap (VREFINT) every conversion, compute the true VDDA, and rescale VBAT_DIV/VBUS_DIV** (the standard STM32/PY32 technique); this recovers the ATtiny's fixed-reference accuracy. **Accuracy ceiling (datasheet Rev 1.7 Table 5-25):** VREFINT is 1.17–1.23 V with **no per-part cal constant**, so the *absolute* result is ~±2.5% (≈ ±75 mV at the 3.0 V cutoff). The sampling's job is to recover VDDA *sag* — the failure mode this bullet warns about — not to add sub-percent absolute accuracy; ±2.5% is adequate for the 0.1 V-granular voltage thresholds, and the relative trend (tracking a cell draining) is much tighter. The 1M/330k divider (below) is unchanged. *The ATtiny reasoning in this bullet stands and is the rollback target if a well-stocked AVR ever returns — see the MCU note at the cell-bottom trace below.*
   - **It makes the STEMMA-QT connector honest** rather than merely the right shape — that ecosystem is 3.3V by convention, and PowerMOD now matches it in fact.
4. **⚠️ Consequence: a 5V host does not merely fail — it damages PowerMOD.** 5V on a pin with a 3.8V absolute maximum. **5V hosts require an external level shifter**, and the Scope section's host claims are narrowed accordingly (below). ### What the CC resistors do to the port-swap risk — better than "accepted", short of "solved" (audit finding, 2026-07-15)

**The port-swap decision was made without CC resistors in the design, and adding them changes its conclusion.** The Interface section accepted the swap risk on silkscreen alone, judging that "backfeeding into the boost converter's output (charger into the output port) is the dangerous direction." **Correct CC implementation makes most of that hazard electrically impossible**, because of a USB-C rule the original analysis had no reason to consider: **a source only energises VBUS once it detects a sink's Rd pulldown.**

| Mis-plug | What CC sees | Outcome |
|---|---|---|
| Compliant **C-to-C charger → output port** | charger's Rp vs. our Rp | No Rd anywhere → **the charger never turns VBUS on. Nothing happens.** |
| **Host → input port** | host's Rd vs. our Rd | No source → no-op, as already documented. |
| **USB-A charger + A-to-C cable → output port** | USB-A has no CC gating at all | **5V is live on VBUS regardless → still backfeeds.** |

- **The residual risk is USB-A sources, and that is the common case** — most people own a USB-A charger and an A-to-C cable, and a legacy USB-A port drives VBUS unconditionally. So the hazard is narrowed to one scenario rather than eliminated. **Silkscreen still earns its place; it is just now guarding a much smaller door.**
- **The genuinely dangerous version is narrower still, and worth naming: 3.3V jumper mode + a USB-A backfeed puts 5V onto a 3.3V host.** In 5V mode a backfeed is 5V meeting 5V — benign, and TPS63020's load disconnect blocks VOUT→VIN while off anyway. **In 3.3V mode it is an overvoltage event on someone else's hardware.** This is the case to weigh if reverse-protection is ever revisited.
- **Note the shape of this, because it is the inverse of the usual finding:** the CC resistors were added purely to make the board *work*, and they turn out to also mitigate a hazard the spec had given up on. The original port-swap analysis wasn't wrong — it was reasoning about a board whose CC pins were unconnected, which is the board the spec described.

### USB-C current advertisement — PowerMOD's output current does not fit the spec's quantisation (audit finding)

**Rp is quantised: Default (500/900mA), 1.5A, or 3.0A. There is no ~1.2A option**, and ~1.2A is what the power budget actually allows (Section on power stage).

> **Re-checked after the charger reversal (2026-07-15) — the decision holds, but its reasoning was wrong and had to be rebuilt.** Output is no longer a single number: it is **~2.7A on mains and ~1.0–1.5A on battery through PH2.0**. So "~1.2A is what the budget allows" is simply false now, and on mains alone **1.5A (Rp=22k) would finally be honest**.
>
> **It still fails, for a better reason.** An advertisement is a *guarantee*, and it cannot be conditional on which source happens to be connected — a compliant sink reads Rp once and draws accordingly, including at 3am on battery with the cell near cutoff, where PowerMOD delivers **~1.0A**. **Advertising 1.5A would promise, in the worst configuration, half again more than the hardware can supply — and the worst configuration is the one a UPS exists for.**
>
> **Rp = 56k (Default) stands.** Note what changed: the old reasoning would have *flipped* on the new numbers, because it compared the advertisement against a typical figure. The rebuilt reasoning compares it against the **floor**, which is the only comparison an advertisement can survive.
- **Advertising 1.5A (Rp=22k) over-promises.** A compliant sink may then draw 1.5A without any negotiation — more than PowerMOD can deliver, producing brownout at exactly the moment of peak demand.
- **Advertising Default (Rp=56k) under-promises**, capping a compliant sink at 500–900mA against a real capability of ~1.0–1.35A.
- **Resolution: advertise Default (Rp=56kΩ), and accept the under-promise.** Over-advertising is a brownout; under-advertising is a limitation. **In practice it costs almost nothing for the documented hosts:** the flagship A64 takes power through a dumb USB-C-to-barrel adapter that ignores CC entirely, and Pi-class boards draw what they need without negotiating. **The advertisement matters only for strictly compliant sinks, which are not the target — but under-promising is the only honest way to be wrong.** Confirm exact Rp values against the USB-C spec during schematic work.

### USB-C input: PowerMOD ignores what the source advertises, and this must be a documented requirement

> **Re-examined after the charger reversal (2026-07-15). Every conclusion below survives; the mechanism behind each one is gone.** This is the most useful thing in the section, so it is recorded rather than rewritten:
> - **"Draws whatever it wants regardless of the advertisement" — still true, and now more so.** There is no `SEL` pin, no VPCC, no ICLC: **TP4056 + OR has no input current limiting of any kind.** The old text's "up to 1.8A" becomes "up to whatever the source will give," so the **≥2A supply requirement stands and hardens.**
> - **"Graceful failure in LiPo mode" — still true, but NOT the same outcome, and an earlier version of this correction claimed it was.** VPCC no longer exists, and neither does its load-*sharing*: **the OR is winner-take-all — it cannot blend sources, so "the battery covers the shortfall" is MCP73871 ideal-diode language with no referent in this topology.** What actually happens: mild sag → the input still carries everything, firmware backs charging off via `CE`, and the host is insulated by the buck-boost's regulation (not by the battery). Severe sag → a **handover**: the whole load moves to the cell. Host unaffected in both; the symptoms differ — *slow charging* in the first, **battery discharging while plugged in** in the second. See the OR stage for the sustained-body-diode consequence.
> - **"Mains-only mode is *harder*, not more forgiving" — still true, and for the same underlying reason.** With no cell, nothing backfills. The old text's "'if possible' is doing real work in that sentence" applies verbatim to the body diode: no battery, nothing to forward-bias, host browns out.
> - **One property is genuinely lost:** VPCC *throttled charging* to protect the input. The OR does not — it just hands the load to the battery. **So under a weak supply the cell now discharges on mains**, where MCP73871 would have reduced charge current first. `CE`-based firmware arbitration is the intended substitute (see the power-stage section), and it is a substitute, not an equal.
>
> **That a section written entirely around one part's pins survived that part's removal unchanged is worth noticing.** The conclusions were about *the system's behaviour under an undersized supply*, and were never really about `SEL` — the part was the evidence, not the reason. The MCP73871-specific text below is retained on that basis.

**The `SEL` pin forces a choice with no middle** (MCP73871 datasheet — **superseded, retained as the evidence that produced the conclusions above**): **SEL high** = "capable of providing 1.65 (typical) total amperes… limits the input current up to 1.8A"; **SEL low** = "adhering to the current limits governed by the USB specification" (100/500mA via `PROG2`).
- **SEL must be tied high.** SEL low caps input at 500mA → ~2.5W → an output around **0.45A**, which would gut the product.
- **Consequence, stated plainly: PowerMOD presents Rd (so any source turns on) and then draws up to 1.8A regardless of what that source advertised.** It does not sense the source's Rp. This is not USB-C compliant, and it is universal practice for this class of board — but it means **PowerMOD's input requires a 5V supply rated ≥2A, and nothing in this design has ever said so.**
- **The failure is graceful rather than dangerous — but only in LiPo mode, and that asymmetry was missed until VPCC was actually read (2026-07-15).** MCP73871 §3.3: "If the voltage on the IN pin drops to a preset value… due to a limited amount of input current or input source impedance, **the battery charging current is reduced. If possible, further demand from the system is supported by the battery.**"
  - **In LiPo mode this is genuinely graceful, and more so than assumed:** an undersized supply sags, charging throttles, **and the battery covers the shortfall**. The host never notices. The visible symptom is a battery that charges slowly or never reaches full — which is why "why won't my battery charge?" should send an integrator to look at their charger first.
  - **In mains-only mode there is no battery to backfill.** "If possible" is doing real work in that sentence: with no cell fitted, VPCC can only reduce a charge current that is already zero. **An undersized supply then sags until the host browns out.** The ≥2A requirement is therefore *harder* in mains-only mode than in LiPo mode — the opposite of the intuition that the simpler configuration is the more forgiving one.
  - **Configure VPCC deliberately** — its threshold is set by a divider to the VPCC pin (1.23V reference), and the datasheet's own worked example sizes it for a 5V supply. Connecting VPCC to IN *disables* the feature, which for this design would be a mistake.
- **Port-swap risk (input vs. output, both USB-C):** identical connectors on both ports create real risk of a user plugging a charger into the host-output port. This is asymmetric — backfeeding into the boost converter's output (charger into the output port) is the dangerous direction and could plausibly damage the converter or upstream components; the reverse mistake (host into the input port) is just a no-op. **Decision for v1: clear silkscreen labeling only** — no physically distinct connector, no reverse-power protection circuitry. **(Revised 2026-07-15 — the conclusion stands but its premise changed: correct CC resistors make the compliant C-to-C mis-plug electrically impossible, narrowing this risk to USB-A sources only. See "What the CC resistors do to the port-swap risk" below. This analysis was written for a board whose CC pins were unconnected, which is the board this spec described at the time.)** Deliberate tradeoff: USB-C is the universal expectation for power in/out, and USB adapters (including the barrel-jack one already needed for the A64) are cheap and everywhere, so keeping both ports as USB-C maximizes general compatibility at the cost of relying on the installer reading the label correctly, particularly at first install or when a technician revisits the device in the field. Worth revisiting (distinct output connector, or a reverse-protection IC) if this ever causes a real field failure.
- **Raw power output pins — added, alongside USB-C.** A small unpopulated header exposing VOUT + GND directly (same internal rail as the USB-C output, no new circuitry), for hosts without a USB-C power input (many bare-metal MCU dev boards use a VIN pin, barrel jack, or direct header power instead). Precedented by Witty Pi's own unpopulated header (their P3, breaking out VOUT/CATHODE for exactly this purpose). Cheap to add since it's just exposing an existing rail, not new regulation — see the extensibility note below for the real constraint this doesn't solve.

## Power stage topology and voltage flexibility — resolved, adopted for v1

**Decision: buck-boost converter (not plain boost), field-configurable output voltage via jumper/solder-bridge, default-ships-at-5V.**

**Why buck-boost, not plain boost, from v1:** A plain boost converter can only step voltage *up*, which is sufficient for a fixed 5V target (always above the LiPo's 3.0–4.2V range) but cannot support a 3.3V target — the LiPo's voltage range straddles 3.3V, so part of the discharge curve would need step-down behavior a boost converter physically cannot provide. Supporting 3.3V requires a genuinely different topology (buck-boost), not just a different-value boost converter — the supporting passives (inductor, feedback divider, capacitors) are calculated for the specific voltage relationship, so a plain-boost-to-buck-boost change is a real subcircuit redesign, not a component swap.

**Why do this now instead of deferring:** the buck-boost topology's added cost (a small number of extra passives, marginally more board area, more verification effort across buck/boost/transition operating modes) is paid once either way — adopting it now avoids ever having to redesign the power stage later if a 3.3V variant is wanted. Deferring would mean building plain-boost now and buck-boost later, paying the design/verification cost twice instead of once.

**Field-configurable voltage selection, not just a BOM-time choice:** many buck-boost ICs support output voltage selection via a simple strap pin or resistor-divider values, which can be exposed as a physical jumper or solder-bridge rather than requiring different components at manufacturing time. This is adopted for v1 — genuinely low component cost (cents), and the buck-boost IC's configurability makes it close to free once the topology decision above is already made.

**Real risk accepted, consistent with existing precedent:** a field-configurable voltage jumper introduces the same class of risk already accepted for the USB port-swap decision (Interface section) — a user could misconfigure the jumper (e.g., leave it at 5V after repurposing the board for a 3.3V-only host) and damage their host. **Resolved with the same proportionate response already used elsewhere in this spec: clear silkscreen labeling at the jumper location, no additional lockout or auto-detection circuitry.** Ships default at 5V (matching the primary Linux SBC use case). The main added cost of this feature is not componentry but ensuring the jumper's current position is genuinely easy to visually confirm before connecting a host — a documentation/silkscreen-clarity task, not a BOM cost.

**Net effect on "does this work for low-power MCUs" (the question that prompted this):** yes, meaningfully more so than the original plain-boost design — a 3.3V-target host is now a jumper setting, not a redesign. Combined with the raw VOUT/GND header (above) and the already-portable I2C protocol layer, PowerMOD's electrical design now generalizes well beyond the primary Linux SBC target with no compromise to that primary use case.

## Charger and power path — REVERSED 2026-07-15: back to TP4056, behind a discrete PFET OR

> **This section supersedes "Charger and power path — TP4056 replaced with a power-path IC" below, which is retained in full as reasoning history.** The superseded section's *architecture* description is still correct. Its **part-selection conclusion is not**, and the error is instructive enough to keep visible.

**What was wrong.** The audit disqualified TP4056 because "it has no SYS rail — the load hangs permanently off the BAT pin," so load current corrupts charge termination and hot-removal browns out the host. **Every one of those statements is true of TP4056 *alone*, and false of TP4056 behind an OR-ing stage.** With a power OR in front, the load hangs on the merged system node, never on `BAT`. On mains the cell is reverse-biased out of the path and TP4056 sees *only* charge current — clean termination. Off mains, TP4056 isn't charging at all. **Charge and discharge become mutually exclusive by construction**, which dissolves the exact defect the part was rejected for.

**This is the audit's own recurring error, committed by the audit.** A defect belonging to a *topology* was attributed to a *part*, because the words sat next to each other — the same shape as the thermistor/connector, load-switch/power-path, and TCXO findings catalogued throughout this document. The tell was available and missed: this spec's own buck-boost section already named the escape hatch ("*if 3A ever becomes a hard requirement… TP4056 charging the battery off to the side, with ideal-diode/PFET OR-ing*") without noticing it invalidated the disqualification three sections earlier.

**What prompted the re-examination:** the question "how does Witty Pi get away with this?" Their manual answers it directly — *"the battery voltage and 5V from USB-C **merge into the input voltage (VIN)**… Because VUSB-C is 5V and is always higher than VBattery (up to 4.2V), the power input from USB-C connector has higher priority. As long as the power supply is connected to USB-C, **the battery will not be drained**."* That is a voltage-OR, not a power-path IC. They pair it with a TP4056 and an MP3423 boost. **The architecture PowerMOD needs was shipping in the competitor the whole time.**

### The OR stage

**The body diodes do the work; gate drive exists only to remove the 0.7V drop.** A PFET's body diode conducts drain→source, so orienting **source = VSYS on both legs** yields correct behaviour with no gate drive at all — the gates are an efficiency optimisation, not the mechanism.

> ### ⚠️ Corrected 2026-07-15 — the first version of this circuit latched on unplug
>
> **An earlier draft tied Q3's gate directly to VBUS and claimed "failover cannot be broken by a firmware bug, because firmware isn't involved."** The claim was true of Q2. **It was catastrophically false for Q1**, and the circuit had a permanent latch-up. Trace the unplug transient:
>
> 1. **Q1 on ⇒ VBUS and VSYS are shorted** through 9.5mΩ.
> 2. Q2's gate is on VBUS, its source is VSYS ⇒ **Vgs(Q2) ≈ 0 ⇒ Q2's channel stays OFF.**
> 3. Only **Q2's body diode** conducts ⇒ VSYS settles at **VBAT − 0.7V ≈ 3.5V**.
> 4. VBUS = VSYS = **3.5V**, held up by the battery *through Q1*.
> 5. Q3's gate sees 3.5V — far above any NFET threshold ⇒ **Q3 on ⇒ Q1 on.**
> 6. **Stable. Latched, permanently, until power is fully removed.**
>
> **The board keeps running, which is what makes it dangerous.** It runs through Q2's body diode: 0.7V dropped, **~1.4W at 2A and ~2.5W at 3.5A — past the DFN's 2W rating** — with VSYS depressed 0.7V for the rest of the session. Nothing can pull VBUS below the battery, so it never recovers.
>
> **Root cause, stated exactly: VBUS only means "mains is present" while Q1 is off. Turning Q1 on destroys the distinction its own control signal depends on.** This is the same error this document catalogues throughout — **a signal credited with a meaning it only carries under a condition nobody traced** — committed here in our own schematic, in the same section that boasted firmware couldn't break it.
>
> **The near-miss is instructive.** The open-items list flagged "brief Q1/Q2 overlap during plug-in" — the *benign* direction (microjoules, self-limiting). **Unplug is permanent.** Having noticed the transient existed was not the same as having traced it, and the wrong direction got analysed.
>
> **A fixed-threshold detector does not fix it either.** If VBUS sags below a fixed trip point under load, Q1 opens → the load moves to the battery → VBUS recovers unloaded → Q1 closes → sags again: **chatter at kHz while switching 3A**, and an undersized supply is precisely what users plug in. A supervisor's 1–5% hysteresis cannot bound it. **The signal must be VBUS compared against VBAT — a comparison Q1's state cannot corrupt.**

```
   VBUS (USB-C 5V)                         VBAT (cell 3.0-4.2V)
        |                                        |
        |                                        |
     [Q1 PFET]                                [Q2 PFET]
     d=VBUS  s=VSYS                           d=VBAT  s=VSYS
     body diode: VBUS->VSYS                   body diode: VBAT->VSYS
        |                                        |
        |   gate --+-- Rpu 100k -- VSYS          |   gate --+-- VBUS (direct)
        |          |                             |          +-- Rpd 100k -- GND
        |     [Q3 BSS138]  d                     |
        |          |       gate <- MCU GPIO      |     (fully passive)
        |          s = GND  + Rpd 100k to GND    |
        +-------------------+--------------------+
                          VSYS  ->  TPS63020  ->  host

   MCU compares VBUS-ADC vs VBAT-ADC:  VBUS > VBAT + margin  =>  Q3 on => Q1 on
```

- **Q2 (battery leg) — fully passive, and it stays that way.** Gate ties directly to VBUS. Mains present: gate = 5V ≈ source → **off**, body diode reverse-biased (VBAT 4.2V < VSYS ~4.95V) → **the cell is isolated and cannot be back-charged through the FET**. Mains absent: gate pulled to 0 → Vgs = −VBAT → **on**. **No firmware, no threshold, nothing to go wrong.**
- **Q1 (USB leg) — MCU-controlled, because nothing passive works.** Q3 (BSS138) level-shifts a 3.3V GPIO down to Q1's gate. The MCU already has ADC taps on **both** VBUS and VBAT (for `MAINS_PRESENT` and battery telemetry), so the comparison is **free in parts** — and firmware can add a **dwell timer**, which no analog hysteresis can match. This leg is not optional: TP4056's `VCC` sits on VBUS, so backfeed would make it charge the battery *from the battery* — a loop that flattens the cell.
- **The safety-critical direction is still passive; firmware only optimises.** Battery failover (the direction that keeps the host alive) is Q2, and Q2 is dumb silicon. Q1's control is an *efficiency* decision: **Q3's gate carries a pulldown, so the fail-safe default is Q1 OFF** → VBUS reaches VSYS through Q1's body diode at ~0.6–0.7V. **A dead, unprogrammed or hung MCU therefore yields a board that still works, just hot and 0.7V down** — degraded, not dead.
- **New hard requirement: the ATtiny's own watchdog must be enabled.** Q1's control is now firmware-dependent, and the one failure that reintroduces the latch is *"MCU hangs with Q1 on, then mains is unplugged."* The MCU's WDT resets it → GPIO goes high-Z → pulldown → Q1 off → latch broken. **This is distinct from the host watchdog documented elsewhere**, which protects the host and would not help here.
- **Failover is automatic and fast.** Q2's body diode conducts the instant VSYS falls below VBAT−0.7V, carrying the load while the gate slews. The load itself collapses VBUS in microseconds, so the handover is quickest exactly when current is highest.
- **⚠️ Sustained weak supply parks Q2 on its body diode — found 2026-07-15 by a user question, documented rather than fixed.** With a bad supply *still plugged in* and Q1 held off by firmware, VBUS recovers unloaded to ~5V — **and Q2's gate is hardwired to VBUS, so its channel can never enhance while the bad supply stays connected.** The battery carries the entire load through the body diode indefinitely: 0.7V dropped, 0.7W at 1A, 1.4W at 2A — **within AON7407's ~2W rating at connector-class loads, but hot**, with VSYS depressed 0.7V and output capability reduced accordingly. Self-clears the instant the supply is unplugged (VBUS→0 enhances Q2 normally).
  - **Why documented and not fixed:** the escape is MCU control of Q2's gate too — one more GPIO (the budget just went to zero twice) plus a FET, to optimise a state that only exists while the user runs a supply the guide already tells them to replace. **Bounded, self-clearing, fault-only: document it.** If field returns ever show cooked Q2s, this is the known fix and it costs a pin.
  - **The user-facing symptom must be stated as "battery drains while plugged in"** — not "charges slowly," which is the *mild*-sag symptom. The guide carries both, separately.
- **Zero quiescent on battery.** With VBUS at 0V, both gate networks have 0V across them and pass no current. On mains it is ~100µA, which is irrelevant. **The sleep budget is untouched** — this was a live risk, since the whole coin-cell/RTC argument rests on nanoamp-class figures.

**Battery telemetry survives, and beats Witty Pi.** Tap the sense divider on **VBAT, upstream of Q2**, and the cell is readable on mains *and* on battery. Witty Pi only senses the merged VIN, which is why their low-voltage threshold can only act off-mains. **Their blind spot is a layout choice, not inherent to the topology** — worth stating, because "the competitor doesn't do it" was nearly taken as evidence it couldn't be done.

### FET selection — **CLOSED 2026-07-15: AON7407 for both Q1 and Q2**

**The worst case is cruel, and it sets the whole requirement.** Q2's gate drive *is* the cell voltage, so **the lowest cell voltage produces the weakest enhancement and the highest current simultaneously.**

**But the worst case is −3.0V, not −2.5V, and that was worth establishing before shopping.** Below the low-voltage cutoff (~3.0–3.1V) **the host is already powered off** — so there is no high current left to conduct. The binding case is *cell at the cutoff threshold, full load*. Requirement: **≤25mΩ at Vgs = −3.0V, ≥4A, Vds ≥ 20V.**

**Q1's requirement is strictly easier** (it gets a full −5V of gate drive), **so one part serves both** — one BOM line, one footprint, and Q1 inherits margin for free.

> **Consider the field behaviour.** With a fresh cell (−4.2V drive) it half-works. As the cell ages and sags toward cutoff, the FET **progressively turns off**, forcing 3.5A through its body diode: 0.7V drop, ~2.5W in a SOIC-8. **It would pass every bench test and destroy itself months later, in the field, as a function of battery age.** Nothing in the part's headline numbers hints at this — 12A and 3.1W both look generous.
>
> **This is why "specified at Vgs = −2.5V" is a hard requirement rather than a preference.** AO4407A's Rds table starts at −6V. **A part that declines to characterise itself at your operating point is not a conservative choice; it is an unknown one** — and the datasheet is telling you it was never designed for this.

### What is lost, and how it is recovered

**Battery-absent detection was the one genuine casualty**, and it is load-bearing in two places (mode selection; the Battery LED's "Off = absent" versus "fast-blink = critical"). It was previously closed *by pointing at MCP73871's detection feature*. TP4056 has none, and Witty Pi cannot do it either.

**Resolved via TP4056's `CE` pin** (Pin 8, datasheet: *"Pulling the CE pin to low level will put the TP4056 into disable mode"*). **Pull `CE` low and run a ~500ms decay-delta test on VBAT (timing corrected by the BOM walk — the divider-only bleed gives τ≈6.2s, so the test reads a sag, not a collapse). This is a **positive test rather than a threshold guess**, and therefore strictly better than what it replaces: it distinguishes **absent** from **deeply discharged**, which was the original requirement the ADC-threshold approach failed. The same pin doubles as a **firmware VPCC** — watch VBUS sag, drop `CE`, charging backs off. **One GPIO closes both.**

**Also lost: hardware input-current limiting.** No VPCC, no ICLC — PowerMOD will overdraw a 500mA source. `CE` is a firmware approximation, not the real thing. Note the failure is still graceful: an overloaded source sags VBUS below VBAT, Q2's body diode conducts, and the battery carries the load. **The cell then discharges on mains under overload** — the one case where the "battery is never drained on mains" property does not hold, and it must be documented as such rather than claimed absolutely.

**Pin budget: net +2, and it consumed the entire margin.** `CHRG`/`STDBY` replace `STAT1`/`STAT2`; a VBUS tap replaces `PG`; **`CE` is new, and so is the Q1 gate GPIO** (the OR's latch fix — see the OR stage). That is 6 against the incumbent's 4. **This was written as "net +1, and it fits" before the latch was found**, on the same day. **The 1616's 2 spare pins are now 0, and the recommendation is the ATtiny1617** — see the pin budget.

Bench item: scope the handover once, to confirm the trace, not to find the design.

## Charger and power path — TP4056 replaced with a power-path IC (audit finding, adopted) — **SUPERSEDED 2026-07-15, see above**

> **Retained as reasoning history. The architecture analysis holds; the part-selection conclusion was reversed** — see "Charger and power path — REVERSED" above. The MCP73871-specific findings below (`THERM` bias, `SEL`, VPCC, STAT decoding) remain accurate *about that part* and are the reason the reversal is documented rather than silently rewritten.

**This is the most consequential change from the audit, and the only one that spends BOM cost rather than saving it.**

**The problem: the speced BOM could not deliver the architecture this document describes.** Section 1.1 and the LiPo hot-removal decision both describe mains taking priority, the battery *only charging and powering nothing* while mains is present, and hot-removal therefore being safe. That is a textbook description of **power-path management**. The TP4056-class charger originally speced does not do power-path management. It is a bare linear charger with no separate system rail — the load connects to the BAT pin, permanently in parallel with the battery. Three consequences, all real:

1. **The battery is never actually isolated from the system rail.** The central architectural claim was false, not merely imprecise.
2. **Load current corrupts charge termination.** This is the well-known "TP4056 with a load attached" failure — the charger cannot distinguish load current from charge current, so termination never triggers cleanly.
3. **Hot-removing the LiPo on mains would brown out the host**, dropping it onto the charger's current-limited BAT output. The documented safety claim was not just unsupported; it was the opposite of the hardware's behavior.

**Resolved: adopt the MCP73871 specifically — not "a power-path charger IC" generically.** This originally said "BQ24074 / MCP73871 class," as though the two were interchangeable. **They are not, and datasheet verification disqualified the BQ24074** (see below). One part closes three separate findings from this audit, which is what justifies the cost:

- **Power-path isolation** — the architecture and the hot-removal safety claim become true, as their own core function rather than an emergent side effect.
- **Input current limiting (DPM)** — resolves the power-budget failure below.
- **Battery-absent detection** — resolves the mains-only mode detection gap (Section on Scope).

**Correction — the separate load switch is *not* subsumed by this part.** An earlier audit pass claimed the power-path charger "replaces the separate load switch listed in the old BOM line." **Wrong attribution:** MCP73871's power path arbitrates *battery vs. mains* upstream; it has nothing to do with switching the *host output*, which is what the old BOM's load switch was for. **The load switch is subsumed by the buck-boost converter instead** — via its 4-switch load-disconnect (see the power stage section) — and only if the chosen converter has that feature. Same error class as the rest of this audit: a capability credited to the wrong part because both live near "power."

### Part selection: MCP73871, BQ24074 disqualified — verified against datasheets (2026-07-15)

The "BQ24074/MCP73871-class" phrasing implied these parts were interchangeable. **They differ on a capability this design depends on**, and the difference only surfaced when the datasheets were actually read rather than recalled.

**BQ24074 — disqualified. Battery-absent detection is not observable.** Its Table 9-2 gives `CHG` = high-impedance for *all* of: charging done, recharging after termination, IC disabled or no valid input power, **and battery absent**. The detection routine is real but purely internal (§9.3.5.4 pulls `IBAT(DET)` and watches `VBAT` to decide whether to charge); the conclusion never reaches a pin. **"No battery" is indistinguishable from "battery full"** — which is exactly the state PowerMOD must tell apart to select mains-only mode and to drive the Battery LED.
- **A second BQ24074 problem found in the same table:** `CHG` goes low **only for the first charge cycle** — "all refresh (subsequent) charges will result in the CHG pin off." So charging status would also be unreliable, breaking the Battery LED's charging-vs-full distinction independently of the battery-detect issue.

**MCP73871 — selected. Battery-absent *is* observable**, and the important discrimination works:

| State | STAT1 | STAT2 | PG |
|---|---|---|---|
| Charge Complete – Standby | High-Z | **L** | L |
| **No Battery Present** | High-Z | **High-Z** | L |

- **The two collide only with the Shutdown rows.** `Shutdown (CE = L)` is eliminated by tying CE high (charging always enabled). `Shutdown (VDD = IN)` is a UVLO condition — the part enters it when the input falls to within ~100mV of `VBAT`, which cannot occur with valid 5V mains and a ≤4.2V cell. **In PowerMOD's configuration `(High-Z, High-Z, L)` uniquely means no battery**; confirm this reasoning survives schematic review.
- **Charging status is reliable** here — Preconditioning / Constant Current / Constant Voltage all assert `STAT1 = L`, with no first-cycle-only caveat.
- **Two further claims verified in the vendor's own feature list:** "Low-Loss Power-Path Management with Ideal Diode Operation" and "Voltage Proportional Current Control (VPCC) ensures system load has priority over Li-Ion battery charge current" — the second is precisely the host-priority behaviour the power-budget resolution depends on, confirmed rather than assumed.
- **Input current:** "Maximum 1.8A Total Input Current Control," `ILIMIT_AC` = 1500 / 1650 / 1800 mA (min/typ/max) at SEL = High. The ~1.5A derate below is therefore sound; design to the 1500mA minimum, not the typical.
- **Charge current:** resistor-programmable 50mA–1A. **Charge voltage is preset by part number** (4.10 / 4.20 / 4.35 / 4.40V) rather than adjustable — order the **4.20V** variant for standard 3.7V-nominal cells.

**MCP73871 `THERM` pin — must be biased with a fixed resistor, not tied off. Board-does-not-charge risk (datasheet-verified).** Since no thermistor is fitted (see the charging-below-freezing section), `THERM` needs deliberate handling: an internal 50µA source biases the pin, and the part "immediately suspends the charge cycle" if the voltage leaves a 0.25V–1.24V window.
- **Floating** → 50µA into an open circuit → above 1.24V → **charging suspended permanently.**
- **Grounded** → 0V → below 0.25V → **charging suspended permanently.**
- **Required: a fixed resistor from `THERM` to VSS** placing the pin mid-window — ~10kΩ gives 0.5V (50µA × 10k), the same value the part expects from the 10kΩ NTC it is designed around. Confirm against the applications section during schematic work.
- **Note how this nearly went wrong:** an earlier draft of this spec said the thermistor input is "tied off," which reads naturally as *tie to ground* — and grounding is exactly what **TP4056's** datasheet prescribes ("The temperature sense function can be disabled by grounding the TEMP pin"). **The same instruction that disables TP4056's sensing permanently disables MCP73871's charging.** Carrying one part's idiom onto another is the same error that produced the battery-detect and TCXO failures; it is recorded here because it is cheap on paper and fatal on a board — the same category as the USB-C CC resistors.

**Method note, because it generalises:** a web summary of this same MCP73871 table reported "No Battery Present" and "Standby" as *identical* states. **The datasheet says they differ** — and the whole part-selection decision turns on that one cell. Search summaries were wrong where the primary source was right; only the datasheet settles it.

### TP4056 architecture claim — verified 2026-07-15, holds completely

The finding that drove this entire charger replacement was checked against the TP4056 datasheet. **All of it stands:**
- **No system rail exists.** Eight pins: `TEMP`, `PROG`, `GND`, `VCC`, `BAT`, `STDBY`, `CHRG`, `CE`. There is no `SYS`/`OUT` output; `BAT` "provides charge current to the battery and provides regulation voltage of 4.2V" and is the only rail a load could attach to.
- **C/10 termination is corrupted by load current, by the vendor's own mechanism:** the part "automatically terminates the charge cycle when the charge current drops to 1/10th the programmed value." Termination senses BAT-pin current, so a load on BAT holds it above C/10 and termination never fires.
- **Battery hot-removal is *worse* than this spec argued, and TP4056's own indicator table documents it:** with "BAT PIN Connect 10u Capacitance; No battery," the status LEDs show "Red LED Coruscate T=1-4 S" — the charger **oscillating on a capacitor-only rail with a 1–4 second period.** Not merely a brownout: a host would be dropped onto a rail cycling every few seconds. The claim in this document was, if anything, understated.
- **Bonus corroboration for a separate decision:** TP4056's TEMP pin is described as "Connecting TEMP pin to NTC thermistor's output **in Lithium ion battery pack**" — a second, independent vendor stating that the thermistor belongs *inside the pack*, which a 2-pin connector cannot carry. The charging-below-freezing withdrawal now rests on two datasheets rather than one line of reasoning.

> **Honesty note (revised down from four findings).** This originally counted the charger's TS pin — temperature-qualified charging — as a fourth win. **That has been withdrawn**: the 2-pin battery connector gives the TS pin no path to the cell (see the charging-below-freezing section below), so the pin buys nothing here and is instead biased inert by a fixed resistor (*not* tied off — see Part selection). The remaining three still justify the part comfortably, but the case is genuinely one item weaker than first argued, and is recorded that way rather than quietly restated as three.

**Power budget — did not close, now resolved by the same part.** Non-PD USB-C at 5V tops out at 3A / 15W. The design speced 3A out to the host **plus** 1A of charging drawn from that same 15W input — oversubscribed on its face. TP4056 has no input-current-limit or DPM feature, so it would have pulled its full programmed charge current regardless and sagged the shared input. A power-path IC throttles charging dynamically when the input can't support both, which is exactly the required behavior: **the host is prioritized and charging absorbs the shortfall.**

**Output current derated: 3A → ~1.2A.** Two independent limits cap this, and **the tighter one is the battery connector, not the charger** — which is not where the derate was originally attributed. MCP73871 limits input current to 1500mA (minimum spec), while JST PH's 2A rating binds harder than that for most of the discharge curve, since a fixed current limit delivers less power as the cell voltage falls. This is a real reduction and worth being honest about: it comfortably covers typical Linux SBC draw and essentially all MCU hosts, but not 3A peaks. Two things made the old 3A figure untenable regardless:

- **The 15W input budget never supported 3A-plus-charging** regardless of which charger was fitted.
- **The PH2.0 battery connector is rated ~2A**, while 3A at 5V from a 3.7V cell is ~4.5A on the battery side — more than double its rating. The old spec contained this contradiction internally, and had already brushed against it: the Witty Pi review notes their manual recommending direct solder pads "for high-current setups," which is precisely this problem surfacing in someone else's product.

**Both limits are now datasheet-confirmed (2026-07-15), and together they land *below* 1.5A.** MCP73871's input limit is 1500mA minimum; JST PH is rated exactly 2A. Efficiency figures below are **read off a representative candidate's curves** (TPS63020, Figure 13: efficiency vs input voltage at Vout 4.5V — the nearest documented operating point to PowerMOD's 5V), minus ~2 points because 5V out is a harder boost ratio than the plotted 4.5V:

| Binding constraint | Efficiency | Output at 5V |
|---|---|---|
| **Mains mode** — charger input limit, 1.5A @ 5V | ~92% (Vin≈Vout) | ~1.38 A |
| **Battery mode** — PH 2.0's 2A, cell at 4.2V (full) | ~90% | ~1.51 A |
| **Battery mode** — PH 2.0's 2A, cell at 3.7V (nominal) | ~88% | ~1.30 A |
| **Battery mode** — PH 2.0's 2A, cell at 3.0V (near cutoff) | ~85% | **~1.02 A** |

- **The two constraints converge on ~1.1–1.35A, not 1.5A.** The earlier ~1.5A figure came from the charger's input limit *alone* and ignored the battery-side connector entirely — which is the tighter constraint for most of the discharge curve.
- **Output capability falls as the cell discharges**, because a fixed 2A connector limit delivers less power at lower cell voltage. That is physically unavoidable, and it means the *guaranteed* figure is the worst case (~1.08A near cutoff), not the best (~1.51A at full charge).
- **Working figure for the BOM: ~1.2A, with ~1.0A as the honest guaranteed floor in LiPo mode.** Comfortably covers typical Linux SBC draw and any MCU host.
- If a higher figure is ever needed, **the connector is the thing to change** (solder pads, already noted above) — not the charger, and definitely not the converter.

**On the efficiency figure itself — this is a different kind of open item from the rest, and worth being precise about.** An earlier draft used a flat ~90% and flagged it as "unverified, must be confirmed." **It cannot be verified, because no buck-boost part has been selected** — there is no datasheet to read. It is a *planning assumption for a component that does not yet exist*, not an unchecked claim about a real one, and no amount of looking will close it until a part is chosen.

What *can* be said, from checking a representative candidate (TPS63020) at PowerMOD's real operating points:
- **A flat 90% was mildly optimistic at the low end.** Efficiency falls as the cell discharges — precisely when the connector limit is already tightest — so the two effects compound at the worst case rather than cancelling. Realistic: ~85% near cutoff, ~88% nominal, ~90–92% full or on mains.
- **The conclusions survive anyway.** Using realistic figures moves the worst case from ~1.08A to ~1.02A — **60mA**. The ~1.2A working figure and the ~1.0A guaranteed floor both hold. This assumption was not load-bearing in the way the failed claims were.
- **The buck-boost is not the bottleneck and never was.** A representative candidate carries 4A switches and delivers up to 3A. The ~1.2A ceiling is set entirely by the charger's input limit and the battery connector — **so the converter can be selected on cost, quiescent current and package, not on current capability.** That widens the BOM options considerably.
- **Quiescent is not a problem either:** the candidate specs 25µA operating quiescent, comfortably inside the ~0.3mA board budget claimed in the Scope table.
- **One real schematic requirement surfaced:** parts in this class have a power-save mode that must be *explicitly enabled* (TPS63020: tie `PS/SYNC` low). With it disabled, light-load efficiency collapses — the datasheet's own curves show single-digit percentages at ~1mA, versus ~80% with power-save on. It costs nothing and matters below roughly 200mA. **Whatever converter is chosen, confirm how its power-save mode is enabled and enable it** — same category as the USB-C CC resistors and the MCP73871 `THERM` bias: free, easy to miss, and it degrades the product silently.

### Load disconnect is a hard requirement, and it is how host power is switched at all

**This was never stated anywhere, and it very nearly went unnoticed.** PowerMOD's entire purpose is cutting power to the host — but nothing in this document ever said *by what mechanism*. The answer is the converter's enable pin, and **that only works if the converter truly disconnects its output.**

- **A plain boost converter cannot do this.** Its high-side rectifier presents a body-diode path from input to output, so disabling it leaves the output sitting at roughly Vin − 0.5V. **Disabling a boost does not cut host power.** That is precisely why the original BOM listed a separate load switch.
- **A buck-boost can, and TPS63020 says so explicitly:** *"Due to the 4-switch topology, the load is always disconnected from the input during shutdown of the converter"*, and *"In shutdown mode… the load is disconnected from the input. This means that the output voltage can drop below the input voltage during shutdown."*

**This independently justifies the buck-boost decision on a ground this document never argued.** The topology was adopted solely because "the LiPo's range straddles 3.3V, so a 3.3V target needs step-down." True — but incomplete. **The 4-switch topology also delivers load disconnect for free, which the whole product depends on.** Even a 5V-only PowerMOD with no 3.3V jumper would need either this or a separate load switch. The decision was more right than the reasoning given for it.

**Consequence for any cheaper alternative: load disconnect is non-negotiable and many cheap boost/buck-boost parts lack it.** A part that saves a few cents and then cannot cut host power has not saved anything — it has bought a load switch.

**On cost, honestly.** TPS63020 is a premium TI part and **not the cheapest possible answer.** It is selected because it is the only candidate whose datasheet has actually been read against PowerMOD's real operating points — its efficiency curves at our Vin/Vout/Iout, its 4A limit, its load disconnect, its 25µA quiescent. **Surveying cheaper parts is a legitimate BOM-stage exercise and should happen** — but every candidate must be checked against the full table above, *especially* the ≥3A switch and load disconnect, and **selecting one from memory would repeat exactly the error this audit exists to correct.** Treat TPS63020 as the verified baseline that a cheaper part must beat on cost *without* failing any row.

- **What remains genuinely open:** the real efficiency of the chosen part, measurable on hardware. Until then ~85–88% is a defensible planning band rather than a guess.

**If 3A ever becomes a hard requirement**, the alternative is a discrete power path — TP4056 charging the battery off to the side, with ideal-diode/PFET OR-ing giving mains priority and true isolation. That preserves high current for a few cents of MOSFETs, at the cost of more discrete design and verification across the buck/boost/transition modes, and it still requires solving charge throttling and the battery-connector rating separately. **Not adopted**, because no documented host needs 3A and the integrated part closes three findings that the discrete route would leave open.

## MCU pin budget — counted 2026-07-15, then re-counted twice the same day. **A 14-pin ATtiny does not fit. The 20-pin part is the floor; the ATtiny1617 (24-pin) is selected.**

Previously carried as "adds up close to what an ATtiny-class part offers, so confirm the count." Counted properly, it doesn't add up *close* — it misses by four.

| Signal | Pins | Note |
|---|---|---|
| I2C **slave** to host (hardware TWI) | 2 | required |
| I2C **master** to the RTC (second, bit-banged bus) | 2 | **required — see below** |
| Button | 1 | required |
| Power LED (bicolor) | 2 | required by the 4-state LED scheme |
| Battery LED (bicolor) | 2 | required by the 4-state LED scheme |

| TP4056 `CHRG` | 1 | charging — **replaces `STAT1`** |

| TP4056 `CE` | 1 | **new, and it earns its pin twice**: battery-absent test *and* firmware VPCC |

| Part | I/O | Free (UPDI reserved) | Verdict |
|---|---|---|---|
| ATtiny1614 (14-pin) | 12 | 11 | **Does not fit — short 6** |
| ATtiny1616 (20-pin) | 18 | 17 | **Fits with ZERO spare** — and only by treating the RTC `INT` as optional |
| **ATtiny1617 (24-pin VQFN 4×4)** | 22 | 21 | ✅ **SELECTED — fits, 4 spare** |

**What drives it: I2C costs 4 pins, not 2 — a datasheet finding, not an estimate.** PowerMOD must be an I2C **slave** to the host *and* an I2C **master** to the RTC. ATtiny1614/1616/1617 has **one TWI**, and its only "dual" feature is *dual address match* (one slave answering two addresses) — **not** dual mode with independent master and slave pins. So the hardware TWI serves the host and the RTC needs a **second, bit-banged bus on two more GPIOs.** That single fact is the whole difference between fitting and not.
- **Sharing one bus is not the escape hatch.** Putting the RTC on the host's bus would save those 2 pins but requires multi-master arbitration — which this document explicitly declares unsupported ("one controlling host", state-machine details) — and would expose the RTC directly to the host, dissolving the abstraction that PowerMOD owns its own state.

**All three charger status pins are genuinely required — none can be dropped.** They are the price of the capability that disqualified the BQ24074, and the price is real:
- `STAT1` + `STAT2` together separate charging `(L, Hi-Z)` from charge-complete `(Hi-Z, L)` from no-battery `(Hi-Z, Hi-Z)`.
- **`PG` cannot be inferred from them:** "No Input Power" is `(Hi-Z, Hi-Z, Hi-Z)` and "No Battery Present" is `(Hi-Z, Hi-Z, L)` — **identical except for `PG`.** Drop it and those two states merge.

> **Re-counted 2026-07-15 after the charger reversal — the conclusion is unchanged, and that is the point.** TP4056 + PFET OR needs `CHRG`, `STDBY`, `CE`, a VBUS tap and the VBAT ADC: **5 signals against MCP73871's 4**, so the total rises 16 → 17 (15 → 16 excluding UPDI) and the 1616's margin narrows from 2 spare to **1**. **The 20-pin decision survives; a 14-pin part is now short 5 rather than 4.** The paragraph above — "all three charger status pins are genuinely required… the price of the capability that disqualified the BQ24074" — is **no longer the live reasoning**, since those pins belong to a superseded part. It is retained because the *conclusion* it drove (20-pin minimum) turned out to be robust against a complete change of charger, which is worth more than the argument itself.

**RESOLVED 2026-07-15: ATtiny1617, 24-pin VQFN 4×4. The 1616 is the floor, not the choice.**

**The margin has been eaten twice in one day, by two unrelated findings, and that is the argument.** The count went 16 → 17 (charger reversal: `CE` + VBUS tap) → 18 (the OR's latch fix needs a GPIO for Q1). **A 20-pin part now fits with zero spare, and only by treating the RTC `INT` as optional.** Neither pin was foreseeable that morning.

**Zero spare is not a design, it is a bet that nothing else will be found** — on a board where two more pins were discovered between breakfast and dinner. The 1617 costs a modest unit-price delta and 24 pins of QFN instead of 20; **the alternative is respinning the board the next time a transient gets traced properly.**

**The package delta is *still negative* at 24 pins — re-checked against the datasheet after selecting the 1617, not assumed.** ATtiny1617 is **24-pin VQFN 4×4 mm (16 mm²)**, against ATtiny1614's 14-pin SOIC150 at roughly 8.7 × 3.9 mm (**33.7 mm²**). **The 24-pin part is 2.1× *smaller* than the 14-pin part** — ten more pins and less than half the board area. The argument was originally made for the 1616's VQFN 3×3 (9 mm²); **going to 24 pins costs 7 mm² and does not come close to overturning it.**

**One option is genuinely lost: the ATtiny1617 is VQFN-only — there is no SOIC variant** (the 1616 offers SOIC-20; the 1617 does not). **That removes any hand-soldered prototype path for the MCU.** It changes nothing in practice — TPS63020 (SON), AON7407 (DFN) and the RV-3028 already made this board reflow-only — but it is a door that closes rather than one that was already shut. The real costs are a modest unit-price delta and committing to QFN assembly (no hand-soldering), which the MCP73871's own package already implies. **The "we need a bigger MCU" instinct is wrong here: this is a smaller board, not a larger one.**

**Note for the deferred temperature-compensation option — DEAD as of the RTC reversal (2026-07-15):** the ATtiny's die sensor is still free, but the BM8563 has **no offset register to correct** — the option's target is gone. Firmware could still adjust *wake times* mathematically; it cannot trim the free-running clock. (Historically: it needed **zero pins** — the sensor is on the ATtiny die. It stayed free even against the 1616's thin margin, which is part of why it remains a firmware decision rather than a board one.

## Charging below freezing — real hazard, deliberately NOT handled by PowerMOD (raised by audit, adopted, then reversed on review)

**The hazard is real and worth stating plainly: charging a lithium cell below 0°C causes lithium plating — permanent capacity loss, and a genuine safety issue rather than merely degraded performance.** Charging above roughly 45°C is separately damaging. The flagship deployment (outdoor, remote, unattended, solar) will charge below freezing, repeatedly, with nobody present.

**The audit originally made this an active requirement**, on the reasoning that it cost "roughly a dime — one thermistor on the charger IC's TS pin," with qualification enforced in hardware and the pin already paid for. **That costing was wrong, and the requirement is withdrawn.**

**Why it doesn't work here — the connector, not the sensor.** The TS pin exists for a specific arrangement: an NTC **inside the battery pack, bonded to the cell**, exposed as a third wire. That is why consumer packs are `+ / − / T`. **PowerMOD has a PH2.0 2-pin battery connector — plus and minus only.** There is no third wire, so the thermistor cannot ride the battery cable. The original requirement said "one thermistor, placed in thermal contact with the battery" without ever specifying how it gets there, and the connector already chosen makes the obvious answer impossible. The thermistor is 10¢; **getting it onto the cell is a connector, mechanical, and user-instruction problem** that the BOM-line framing hid entirely.

**Why the tempting fallback is worse than nothing.** Putting the NTC on the PCB does not merely *approximate* cell temperature — **it errs in the dangerous direction.** During charging the board self-heats (buck-boost and charger both dissipating), so the PCB reads warmer than a cell on a wire a few centimetres away in the same enclosure. Reading +5°C while the cell sits at −5°C permits exactly the charge that plates lithium. A protection feature that fails toward "yes, proceed" is worse than no feature, because it manufactures confidence the hazard is handled.

**Why the alternatives were rejected:**
- **3-pin battery connector requiring an NTC-equipped pack** would give real, automatic protection — but a 2-wire pouch physically cannot mate with a 3-pin housing, so it narrows supported batteries to NTC packs and breaks the "works with any 3.7V-nominal Li-ion or LiPo battery" promise outright. That is a product repositioning, not a connector change.
- **Optional external NTC on its own header**, bonded by the installer, preserves "any LiPo" — but protection then depends on a user assembly step most will skip, and does nothing if omitted or left dangling. A feature that exists in the documentation and not in the field is arguably the worst outcome of the three: everyone believes cold charging is handled and almost no deployment actually is.

**Resolved: PowerMOD does not qualify charging by temperature. No thermistor is fitted — the charger's `THERM` pin is biased mid-window by a fixed resistor so charging is permitted unconditionally (see Part selection; "tie it off" is *wrong* for this part and would suspend charging forever) — and the hazard is documented as the integrator's to manage** — consistent with the "push responsibility to the user/integrator" philosophy applied throughout, and with Witty Pi, which has no charge-temperature qualification either. **This leaves the outdoor/solar case genuinely exposed, and that is stated rather than papered over:** an integrator deploying below freezing must handle it themselves (a cell rated for low-temperature charge, an enclosure that stays above zero, or host-side logic that declines to run when it's too cold). **Worth revisiting only if a variant with a defined battery — rather than "any LiPo" — is ever built**, since a known pack is exactly what makes the TS pin usable.

## Connector/mounting footprint — already generalizes well, no further action needed

Unlike Witty Pi's HAT form factor (mechanically dependent on the Pi's specific 40-pin header), PowerMOD's small I2C connector plus separate power connectors were already chosen specifically to avoid this kind of host-specific mechanical dependency (Section 3.4). This is a genuine existing strength, not a gap.

## Explicit scope decisions (minimal-board philosophy)

Consistent principle applied throughout: keep the board simple and stateless beyond the schedule; push responsibility to the host/integrator wherever reasonable. Two items below were originally deferred as low-priority but reopened on review, since the fix is near-free now and expensive to retrofit after boards are deployed.

- **RTC default state:** Unix epoch (1970) on first power-up / after full backup depletion. No "unset" state or refusal to schedule. **Two legitimate usage modes, both fully supported, host's choice:** (1) host periodically syncs real wall-clock time to the RTC (not just once — any time it wants, as often as it wants) if it cares about absolute accuracy; or (2) host never sets wall-clock time at all and relies purely on the relative-time scheduling pattern (read PowerMOD's current value, add an offset, write that back) — in which case what the RTC reads relative to true time is irrelevant, only its own internal consistency matters. Neither mode requires host/user action before scheduling works correctly; only mode (1) requires setting real time if wall-clock accuracy is actually wanted.
- **Behavior when power is first connected — three modes, default changed from Always ON to Restore previous state (audit finding).** Witty Pi defaults to OFF when power is connected (requires a button press or existing schedule to boot the host), configurable via a register. PowerMOD offers three modes via a configurable register:
  1. **Always ON** — boot whenever power is available.
  2. **Always OFF** — require an explicit button press or scheduled wake, matching Witty Pi's default. Preferred by some hobby/bench users who don't want unattended auto-boot.
  3. **Restore previous state (default)** — if the host was unpowered (whether from a host-requested shutdown or a button press) at the moment mains/battery power was lost, stay off when power returns; if the host was actively powered at that moment, boot again. **Note: "previous state" here means PowerMOD's own power-output state (was the host powered or not), not any concept of host-side sleep/suspend — PowerMOD has no visibility into or awareness of host OS power states like Linux suspend-to-RAM.** Cheap to add: one bit of persisted state, stored in **EEPROM** (corrected — see below), reusing existing infrastructure rather than adding new hardware. **An emergency cutoff records the host as ON — it was cut involuntarily — so recovery re-boots it under this mode (semantic review, item 1).**

  **Why the default changed — the argument moved, not just the value.** The original reasoning for defaulting to Always ON was: the primary motivating scenario is remote, unattended deployment where mains may be interrupted and later restored (grid blip, solar reconnect) with nobody present to press a button, so defaulting to OFF would leave the host dead until physically visited. **That reasoning does not survive contact with the rest of this design, for two reasons:**
  - **The recovery it was protecting against already happens without it.** If a wake time was scheduled, it survives in EEPROM, the coin-cell-backed RTC keeps the clock it's measured against, and the missed-alarm rule (Section on fallback safety) fires it as soon as power returns — *in every mode*. Always ON was never what was rescuing that case. It only adds value when the host is off with **no** wake pending at all, which is the abnormal case, and which the opt-in max-sleep safety net already backstops.
  - **Its cost lands squarely on the deployment it was justified by.** Always ON means a 2 AM grid blip boots a host that deliberately went to sleep until 6 AM. A well-written host would notice and go back to sleep; a perfectly ordinary cron-driven host with no "should I be awake right now?" logic will simply sit there and burn four hours of battery. That is the scheduled-power-saving use case — the whole point of the product — defeating itself, unattended, on battery.

  **Restore previous state gets every one of these right:** a host that was deliberately asleep stays asleep through a blip; a host that was actively running when power died resumes; a scheduled wake still fires regardless. It fails only on a total power loss that *also* kills the coin cell — the wake time itself survives in EEPROM, but the clock it was measured against does not, so an absolute schedule becomes meaningless. That is precisely the case the coin cell exists to prevent, and max-sleep backstops it besides.

  **Tradeoff accepted for the new default:** on a genuinely first-ever power-up there is no meaningful "previous state." The persisted bit ships defaulting to ON, so a fresh board still boots the host on first install (preserving the original default's one clear benefit), and the mode only starts meaningfully constraining behavior once the device has actually run. All three modes remain available; this changes which one ships.
- **Protocol versioning — reopened, made an active requirement, and SATISFIED: `PROTOCOL_VERSION` (`0x00`, reads `1`) and `FIRMWARE_REVISION` (`0x01`) shipped in register map v1.** Originally deferred as unnecessary for v1. Reopened: this board is explicitly meant to work across "as many hosts as possible" (multiple integrators, multiple projects, over a long lifetime) — exactly the condition where firmware/protocol drift becomes real, not hypothetical. Cost of adding a single read-only protocol-version register now is near-zero; cost of retrofitting version-awareness onto boards already deployed in the field is real. **Requirement: include a read-only protocol-version register in the register map**, even if v1 firmware is the only version that will ever exist for a while — the host can check it once at integration time and the door stays open for a compatible v2 later.
- **Last-power-off-reason status bit — reopened, now an active requirement.** Originally scoped out as part of "no event log." Reopened: this isn't really a logging feature, it's a single bit of state, and without it there's a real gap — if the board's independent low-battery cutoff (above) ever force-cuts the host, and the board keeps zero record, the operator has no way to tell "the schedule did this" from "the battery is failing and force-cut the host" the next time they check the device. That's the difference between "working as designed" and "battery about to die unattended, come fix this." **Requirement: one readable register indicating whether the most recent power-off was a scheduled/host-requested cut or a forced low-battery cutoff.** This is not a log or history — just the single most recent reason, overwritten each time — so it stays consistent with the "no event history" scope decision below while closing the one gap that actually mattered.
- **Reason codes must be split into two registers — power-off and power-on (audit finding, genuine bug in the prior design).** The requirement above, the detailed reason enum adopted from Witty Pi, and the reason-retention rule below were each individually reasonable and collectively broken. The enum merges power-on reasons (scheduled wake, button) and power-off reasons (scheduled cut, low voltage) into **one** register, and retention was defined as "retained until overwritten by the next event (**scheduled wake**, scheduled shutdown, …)".
  - **The consequence: the power-off reason is always destroyed before any host can read it.** A host must be powered on to read a register. Powering on *is* an event. So the wake overwrites the power-off reason during the very act of booting — and the user guide instructs hosts to check the reason on every boot, which is precisely the moment it has just been clobbered. The single most valuable diagnostic in the product ("was I force-cut by a dying battery?") was unreadable by construction.
  - This was masked by the coarse boolean flag surviving, since it is a separate register only written on shutdown. That is very plausibly *why* Witty Pi ships both a boolean and an enum rather than the enum alone.
  - **Resolved: two independent registers — `last power-off reason` and `last power-on reason`** — each retaining its own most-recent value and each overwritten only by an event of its own kind. Free (one extra byte), and it makes the boolean flag a convenience rather than the only thing that works.
- **Host reboots unexpectedly mid-sequence** (after "wake at T," before "cut now") — resolved (see full reasoning below): a wake alarm that fires while the host is already on is a NOP. Does not weaken the battery safety floor above — over-discharge protection still applies regardless.
- **Event log — superseded, see below.** Originally scoped out entirely (only the single last-power-off-reason register). **This decision was later reversed** after real cost was quantified (see "Event log — reconsidered again, decision reversed, adopted," further down this document) — an 8-entry circular log is now part of the design. This entry is kept for historical context on how the decision evolved, not as the current resolution.
- **No fuel-gauge/state-of-charge estimation.** Standard 3.7V-nominal LiPo conventions only (full ≈4.2V, cutoff ≈3.0–3.3V), raw voltage exposed, hard safety floor as above. Matches Witty Pi's own approach; real coulomb-counting for arbitrary user-supplied cells is out of scope.
- **Decommissioning / long-term storage — out of scope, user's own responsibility.** No board-specific guidance beyond what any LiPo/electronics product would already require the user to know (storage voltage, self-discharge, standard battery handling). Consistent with the "push responsibility to the user/integrator" philosophy applied throughout.
- **Multiple PowerMODs on one host — not a supported/considered use case.** No plausible reason a single host would need more than one PowerMOD; not designing for I2C bus discovery/scanning or any other multi-unit scenario.
- **Invalid/past schedule write — resolved.** No board-side validation of wake-time writes (no rejection, no "is this in the past" checking), regardless of whether the host uses absolute or relative wake times. **Both absolute and relative-time scheduling are fully legitimate**, and PowerMOD doesn't distinguish between them — it's just a timestamp write. Absolute wake times are necessary for inherently wall-clock-based schedules (e.g. a cron-style "run at :15 past every hour") and are fine whenever the host has good reason to trust both PowerMOD's RTC and its own clock (e.g. actively NTP/chronyd-synced). **Relative-time scheduling — computing wake time as "PowerMOD's current RTC value + offset" rather than an independently-calculated absolute timestamp — is recommended specifically for hosts that either don't sync wall-clock time at all (Section on RTC default state, above) or want to eliminate clock-drift/timezone-bug risk entirely**, not as a universal default. A host that chooses absolute scheduling and miscalculates a timestamp (drift, timezone bug) is a host-side integration error, consistent with the "host owns the decision, board doesn't compensate for host mistakes" philosophy — not something PowerMOD attempts to catch.
  - **Behavior for a wake time already in the past — resolved, then found to contradict the missed-alarm rule, now re-resolved as commit-time evaluation (audit finding).** External review proposed "wake immediately" once a stored wake time is recognized as already past. This was **rejected** in favor of: a past wake time is simply never treated as due. Reasoning (still valid): "wake immediately" means a host that writes a bad timestamp (e.g. an integer overflow, a timezone bug) gets an instant, unexplained reboot the moment it finishes its shutdown sequence — indistinguishable from a boot loop, and it doesn't fix the underlying bug, just makes its symptom more disruptive. Treating a past wake time as never-due means the host just stays asleep longer than intended (a recoverable, non-alarming consequence), with the opt-in max-sleep safety net remaining the backstop against indefinite stranding.
    - **The contradiction:** the missed-wake-during-outage decision (Section on fallback safety) independently resolved the *opposite* behavior — fire immediately on power restore — for a stored timestamp that is equally, indistinguishably in the past. **Two rules, opposite behaviors, identical observable state.**
    - **Re-resolved: evaluate at commit time.** Past when the host commits it → expired, never fires (this rule's intent, preserved). Future when committed, but overtaken while PowerMOD had no power to act → fires on power restore (the missed-alarm rule's intent, preserved). See the fallback-safety section for the full reasoning; the two decisions are now one rule with a single implementation.
    - **Still not validation, and the distinction still holds:** PowerMOD performs no rejection and no "is this plausible" checking on the write. It accepts any well-formed timestamp. Commit-time evaluation only decides *what the accepted value means*, which is the same accept-vs-act separation already articulated in the wording-correction section below.
- **Host strands itself with no wake time set (e.g., "cut now" issued, or schedule write silently failed, with no valid wake time ever written) — resolved with a narrow safety net.** Default behavior: PowerMOD sleeps indefinitely, matching the "dead simple" philosophy above — no implicit fallback. **Optional safety net: a configurable maximum-sleep-before-forced-wake duration, disabled by default.** If enabled (e.g., set to 24h), PowerMOD will wake unconditionally after that duration regardless of whether a valid schedule was set, giving the host another chance to establish a proper schedule. Disabled by default so this never surprises an integrator who didn't ask for it; available for anyone deploying to a genuinely unattended/remote context who wants protection against the specific case of a failed or missing schedule write silently stranding the device. Mirrors the existing disable-by-value convention already used for the low-voltage and recovery thresholds.
  - **Voltage-gated like every other automatic wake (spec walk, 2026-07-15):** a max-sleep expiry while the battery sits below the recovery threshold with no mains is **held until power is usable**, same rule and same reasoning as semantic-review finding 3 — the safety net must not boot the host onto a cell that will immediately cut it, consuming nothing but the battery's last margin.
  - **Audit note: this decision is the precedent that carried the watchdog.** This safety net and the opt-in watchdog (Section on fallback safety) are the same mechanism — an opt-in timer that recovers a host which failed to declare something it was supposed to declare — applied to the two halves of the same problem. This one covers a host that never says "wake me"; the watchdog covers a host that never says anything again. Accepting one and refusing the other, as this spec previously did, was an inconsistency rather than a principle. They should be read, configured, and documented as a matched pair.

## Semantic review — five holes found by walking every state, mode, and transition (2026-07-15)

Prompted by the question "do all the states and transitions actually make sense?" — asked *after* the staleness and consistency passes were clean, which is the point: **every one of these was logically wrong while being textually consistent.** No sweep for stale part names or mismatched numbers could have found them; only executing the state machine on paper did.

1. **Emergency-cutoff recovery had no defined re-power semantics — three documents implied three different answers.** The guide's §4.3 implied auto-reboot at recovery ("does not restore power *until*…"), the SVG said "returns to OFF" full stop, and the power-on-mode definition keyed on "the moment power was lost" — **power is never lost in an emergency cutoff; PowerMOD cuts it deliberately and stays alive**, so the governing rule never engaged at all. Meanwhile the recovery threshold's own anti-oscillation rationale (boot current re-sagging a marginal cell) presupposes that recovery causes a boot. **Resolved: an emergency cutoff records the host as ON** (it was cut involuntarily; it never chose to sleep). Recovery-condition-met is a power-available event, applied through the normal power-on mode: Restore-previous and Always-ON boot; Always-OFF waits. Reason code `POWER_RESTORED` covers both the mains and the battery-recovery variant. One rule, and §4.3's behaviour, the mode table, and the diagram all fall out of it.

2. **The Battery LED violated the Power LED's budget argument in the same state.** Below the floor, the Power LED drops to a rare flash *because every milliamp comes out of a cell below its protection voltage* — while the Battery LED's critical state, as written, kept fast-blinking there, spending exactly the budget the faintness protects. **Resolved: below the cutoff the Battery LED goes dark**; the rare flash is the board's only light. The fast blink is the *warning* (above cutoff, below critical); darkness plus rare flash is the *cutoff*. **Amended on user review (2026-07-15): the rare flash belongs on the Battery LED, and the Power LED goes dark.** The first fix obeyed the budget but kept the signal on the wrong indicator — the battery's own LED fell silent at the battery's worst moment while the *power* LED narrated a battery crisis. The budget argument only ever said *one LED may flash*; it never said which. Swapped: one LED now tells the whole discharge story, and a dark Power LED means what it says.

3. **Wakes were not voltage-gated, so the protection floor only protected a running host.** A wake coming due while the cell sat below the cutoff (host asleep — the emergency state is only entered from ON) would boot the host onto a cell below the floor, sag it, cut it, and consume the wake: **stranded, by the safety system.** The guide already promised "as soon as it physically *can*." **Resolved: a due wake while battery < recovery threshold with no mains is held and fires when power is usable, reported as `MISSED_WAKE`** — the enum's meaning extended from "power absent" to "power absent or unusable," which is what it always meant in spirit.

4. **A late watchdog kick did not cancel the watchdog's own shutdown.** A busy-but-alive host missing one deadline got power-cycled even if it kicked one second into the 30s forced window — the watchdog reducing the service it exists to restore. Button-initiated shutdowns already had the symmetric escape (second press). **Resolved: a kick during the watchdog's `DELAY_FORCED` window cancels the shutdown, logged `SHUTDOWN_CANCELLED`.** Costs one firmware branch; uses no new register. The cancel matrix is now: button cancels button-initiated; kick cancels watchdog-initiated; nothing cancels host-requested (the host already declared itself done) or low-battery (safety floor).

5. **Hold-from-OFF was unspecified and its literal reading was absurd** — press from OFF boots the host, keep holding 5s, and the force-cut fires on the host you just booted. **Resolved: the hold gesture exists only when the host was ON at the moment of the press.** From OFF, a press powers on and further holding is inert. Separately pinned: during the factory-reset gesture (held at power-connect), **the host is not powered until the gesture resolves**, so a reset never swaps configuration under a booting host.

### The register map, walked the same way — five more (2026-07-15)

Executing the protocol on paper — every register, every sequence a real host runs, every crash point — after the state-machine walk above. Same species of bug: **each rule was correct inside one transaction and wrong across two.**

6. **The read latch outlived its transaction.** "Released when the pointer leaves the field" plus a persistent I2C pointer meant: read `RTC_TIME`'s first byte, STOP, return minutes later for the rest — and receive **minutes-old latched bytes spliced into a coherent-looking, silently wrong timestamp.** A partial read poisoned a later one. **Resolved: the latch also releases at STOP — it never outlives its transaction.**
7. **The event log latched per-entry, so the *set* could tear.** Newest-first presentation means a new event shifts every entry down one; a bulk read straddling the shift sees an entry twice and misses another, each entry individually coherent. **Resolved: the log latches as one block on the first byte read in it.**
8. **Uncommitted staged config masqueraded as the running config after a soft reboot.** Config reads return *staged* values; PowerMOD survives host crashes, so stale staging lingered — and a rebooted host could not read the active values through it, at all. **Resolved three ways: staging is discarded whenever host power is cut; `CONFIG_STATUS = 5` reports uncommitted staged changes; `CONFIG_COMMIT = 0x44` discards them** — required, because a host detecting the leftovers otherwise had no way out except committing values it never chose.
9. **Factory reset by register (`0xDF`) would have reverted the I2C address mid-session** — disconnecting the host that issued it, before it could even read the result. **Resolved: the address alone follows the standing rule (next full power cycle); everything else reverts immediately.**
10. **`WAKE_ARMED` said "future wake" — and the held wake introduced by finding 3 above is due, not future.** As defined, the bit read `0` while a wake was pending, telling the host nothing would happen right before something did. **Resolved: armed = committed and not yet fired** — future, or due-and-held. "Armed" answers *will something happen*; for a held wake the answer is yes.

Two clauses added in passing: a cancelling watchdog kick restarts the countdown (a late-recovering host isn't re-condemned by the deadline it already missed), and the `CE` battery test runs periodically on mains (millisecond charge pauses; hot-swapped cells noticed without a reboot).

### The spec itself, walked as decisions against decisions — four more (2026-07-15)

The third walk: not states (guide), not transactions (register map), but **whether the decisions compose** — does the mechanism chosen by one decision actually implement the behaviour promised by another?

11. **The restore-previous derivation mechanism contradicted semantic-review finding 1.** The state bit is derived from the event log ("most recent event a power-off ⇒ host was off") — which reads an emergency cutoff as *off*, while finding 1 requires it to count as **ON**. The rule and its mechanism were adopted months apart in document time and never re-met. **Resolved: derive by voluntariness** — deliberate cuts (`HOST_REQUEST`, `BUTTON`, `BUTTON_FORCED`) ⇒ OFF; involuntary (`LOW_VOLTAGE_CUTOFF`, `WATCHDOG_EXPIRY`) ⇒ ON.
12. **Max-sleep was a second ungated path onto a dead cell.** Finding 3 gated *scheduled* wakes below the recovery threshold — and the max-sleep forced wake, "fires unconditionally," was left as a way to do exactly what finding 3 forbids. **Resolved: held identically.** ("Unconditionally" meant "even with no schedule," never "even onto a cell that will cut it.")
13. **The button below the floor was undefined, and its literal reading burned the cell.** A press with the cell below the cutoff would boot the host, trip the low-voltage trigger immediately, and then run the **30-second forced delay below the protection floor** — the cutoff's own grace window spending the charge the cutoff exists to save. **Resolved, two tiers: below the cutoff, the button does not boot** (the floor is as absolute for starting as for running); **between cutoff and recovery it does** — a present human may override into the hysteresis band, which exists to absorb exactly that boot sag. Automatic triggers gate at *recovery*; the human gates at *cutoff*.
14. **The battery-sense divider — the standby floor's largest term — was flagged, never sized, and counted in no floor statement.** The Interface section named the tension (drain vs. ADC source impedance) and left it; every floor claim then quoted "LDO 1µA + sleeping MCU," omitting a ~3µA term. **Resolved: 1M/330k + 100nF sampling cap, ≈3.2µA max, now named in every floor statement.** ~4% of the cell's own self-discharge — harmless, but a floor that omits its largest term isn't a floor.

### The diagram, walked as drawn geometry — no new design holes, one new failure pattern (2026-07-15)

The fourth walk extracted the SVG's actual drawn graph — every box, edge, and label — and diffed it against the post-review ruleset. **The graph itself was sound**: five states, every transition present, no orphan edges, and the earlier reviews' *structural* changes (all four shutdown triggers entering identically, hold-to-force as a separate path, the cancel edge) were already drawn.

**What lagged was every label the three walks had touched.** The cancel label still said button-only (finding 4 added the kick). The trigger list had an unqualified button (finding 13), unqualified alarms and max-sleep (findings 3, 12), an un-annotated Restore-previous (finding 11), and a missed-alarm definition that predated "unusable power." The notes panel lacked the recovery-reboot rule (finding 1). **Ten label edits, verified by rendering** — two of which collided with a drawn edge on the first attempt and were caught by looking at pixels, not markup.

**The pattern, and it inverts the one this document already knows:** the `<desc>` — the diagram's self-description — was *fresher than the diagram*, because three review passes had dutifully updated the prose while nobody re-opened the drawing. Self-descriptions rot when maintenance touches only the artifact; **artifacts rot when maintenance touches only the description.** Both directions come from the same cause — updating whichever layer is cheapest to edit — and the check is the same in both: regenerate one from the other and diff. For this file: extract the drawn labels, compare against the desc, and render before shipping.

**Method note:** findings 1 and 3 are the same species as the audit's hardware findings — a rule credited with coverage it only has under a condition nobody traced ("power was lost" vs. "power was cut"; "the floor protects the host" vs. "the floor protects the *running* host"). The state machine equivalent of the wrong-mechanism error, and the reason this review happened on paper before firmware exists.

**Verified against datasheets in this walk:** TPS63020 `EN` VIH = **1.2V fixed** (3.3V GPIO ⇒ 2.75× margin; VIL 0.4V, so the 100k pulldown holds it off cleanly). TP4056 `CE` accepts "TTL or CMOS logic level" ⇒ 3.3V drive on a 5V-supplied part is in spec. Q2's gate excursion spans **+2.5V to −4.2V**, inside AON7407's tight ±8V limit. *(Historical: the then-selected RV-3028 shipped `TCE=0` and `BSM=00` factory-disabled — the finding that created the delivery-default check; its successor's equivalent is the always-on CLKOUT, below.)*

**Findings — each one a feature that would not have worked as wired:**

1. ~~The VBACKUP pads did nothing without firmware initialization~~ — **this finding was made against the RV-3028 and is superseded by the RTC reversal: the diode-OR needs no enabling, so the failure class it caught is gone.** What replaces it: the BM8563's **CLKOUT ships enabled at 32kHz** (the same delivery-default class) — firmware disables it at init — and supercap charging is a **solder jumper**, not a register. Retained below as written, because the finding is why the walk now checks delivery defaults at all. Original finding: Switchover disabled on delivery means a fitted coin cell would never take over — the flagship backup feature was decorative pending one unwritten EEPROM bit. **Resolved: firmware programs `BSM` (level-switching) unconditionally at first boot** (harmless with nothing fitted, persists in RTC EEPROM), **and a new `BACKUP_TRICKLE` config register (`0x51`, default 0) exposes the trickle charger** — host-controlled because the board cannot know whether the pads hold a supercap (wants charging) or a primary coin cell (charging is a hazard). Default is the safe direction.
2. **TPS63020 `EN` floats during any MCU reset — including the WDT reset this design *requires*.** An undefined `EN` during the ~15ms of an MCU watchdog reset is an undefined host rail. **Resolved: 100kΩ pulldown — and the tradeoff is stated, not hidden: an MCU firmware hang now power-cycles the host** (EN falls, converter disconnects, host caps sag in well under 15ms). The alternative — a pull-up — spuriously *boots* a deliberately sleeping host on every MCU hiccup, which is worse. On re-boot, firmware re-derives state from the event log and restores the host. Rare (the MCU firmware is a few kB), bounded, and now chosen rather than inherited.
3. **Q1's gate never reaches 0V, and nobody had done the voltage division.** The 47k series + 100k pull-up form a divider when Q3 conducts: the gate lands at ~0.32×VSYS, so **Q1 enhancement is Vgs ≈ −0.68×VSYS ≈ −3.4V on mains** — not the full −5V casually assumed. **Adequate by the datasheet** (AON7407 is specified at −2.5V, 12.5mΩ max), so the values stand; recorded because "adequate by luck" and "adequate by check" look identical until the resistor values change.
4. **The 3.3V rail constrains the LED colors, and no document said so.** A blue or white LED (Vf ≈ 3.0V) leaves ~0.3V across its resistor — brightness then swings wildly with Vf spread and rail droop. **Requirement: red/green bicolor packages (Vf ≈ 2.0–2.2V)**, which leave >1V of regulation headroom. Colors A/B in the LED scheme are now red/green, not designer's choice.
5. **The VBUS divider was decided but never sized. Resolved: the same 1MΩ/330kΩ + 100nF as the battery divider — deliberately identical**, because the Q1 decision compares VBUS against VBAT, and identical ratios make the comparison ratio-free: firmware compares raw ADC counts, no scaling, no calibration term. Drains ~4µA only while mains is present, where it is irrelevant.
6. **The internal RTC bus had no pull-ups in any BOM list.** I2C is open-drain; a bit-banged master with no pull-ups works never. 2×10kΩ to 3V3, added.

**Feature-to-path trace, complete:** every register in the map now has verified hardware behind it — RTC (internal bus), battery voltage (VBAT divider), mains presence + input telemetry reservation (VBUS divider), battery-present (`CE` test), charging (`CHRG`), host power (`EN`), backup charging (a solder-jumpered resistor — hardware, deliberately no register since the RTC reversal), event log/config (EEPROM), host watchdog (I2C), MCU watchdog (internal, now with the `EN` pulldown making its reset behavior defined). The two paths that remain unproven by paper are the two already on the bench list: the `CE` collapse test, and converter efficiency.

## 8. Open questions from bulletproofing review

Raised on a systematic pass for failure modes beyond what earlier discussion covered. Witty Pi's own open-source firmware (github.com/uugear/Witty-Pi-4) was checked directly against these — several are now resolved with real precedent, others remain open pending their schematic files.

**Resolved via Witty Pi 4 L3V7's actual user manual (full manual reviewed, not just firmware source):**

- **Protocol/firmware version register — confirmed, with real values.** Register #12, "firmware revision," range 1-255. Adopt the same simple approach.
- **Last-power-off-reason — Witty Pi ships both a coarse flag AND a detailed enum, simultaneously.** Register #8: single boolean, "was previous shutdown due to low voltage." Register #11: full reason enum (0=N/A, 1=ALARM1/startup, 2=ALARM2/shutdown, 3=button click, 4=input voltage too low, 5=input voltage restored, 6=over temperature, 7=below temperature, 8=ALARM1 delayed). **Adopt both patterns** — a quick boolean for simple host logic, plus the full enum for real diagnostics. Confirms the hysteresis/re-arm design (Section 3.5) matches shipped practice: "low voltage" (4) and "voltage restored" (5) are tracked as distinct events.
  - **Adopt the pattern, but not Witty Pi's single-enum structure (audit finding).** Note their enum mixes power-on reasons (1=startup, 3=button click) with power-off reasons (2=shutdown, 4=low voltage) in one register. Copying that structure directly is what produced the overwrite bug documented in the explicit-scope-decisions section above. **PowerMOD splits this into two enums — one for power-on, one for power-off** — while keeping the coarse boolean and the general "flag plus enum" idea intact. The reason-code values themselves can still follow Witty Pi's precedent; only the partitioning changes.
  - **New reason codes added by this audit:** watchdog expiry (power-on) and button-forced-off (power-off, already required below). **No temperature *reason codes*** — Witty Pi's enum reserves over/under-temperature (6, 7); PowerMOD takes no temperature-triggered *actions*, so there is nothing for a reason code to explain. Their board acts on temperature; ours reports it. - **I2C address — confirmed configurable, with the real mechanics.** Default 0x08, configurable by writing register #16 — but changing it requires updating host-side software too and a full power cycle to take effect; it's not a live-changeable value. **Add to register map: configurable I2C address, default 0x08**, and document the same "requires host-side update + power cycle" caveat.
- **Firmware update path — resolved, upgraded beyond documentation-only (external review feedback adopted).** Field-updatable via a programming header + Arduino IDE/programmer. **Correction (2026-07-15): this said "ICSP header," which is wrong for the selected MCU and would send an integrator to buy the wrong tool.** ATtiny1617 is **UPDI-only** — "Single-pin Unified Program and Debug Interface (UPDI)", with *no* SPI-based ISP support anywhere in its datasheet. A conventional 6-pin AVR ISP programmer will not work at all. Practical consequences, all mildly in our favour: the header is **3 pins (UPDI, VCC, GND) rather than 6**, it costs **one MCU pin rather than four** (which the pin budget already assumed), and the tooling is UPDI-capable — SerialUPDI, jtag2updi, or a Microchip debugger — still drivable from the Arduino IDE via megaTinyCore, so the "no exotic toolchain" spirit of this decision survives intact. Confirmed real gotcha via Witty Pi precedent: reflashing resets RTC crystal offset (their register #37) to zero, degrading accuracy until manually restored — Witty Pi's own fix for this is purely documentation (tell the user to record and restore it manually). **PowerMOD improves on this: user configuration registers (voltage thresholds, power-on mode, I2C address, etc.) are stored in EEPROM/flash memory separate from the region firmware programming overwrites**, so a firmware update does not require the user to manually back up and restore this state — it should simply survive. This is a real, small firmware/memory-map design decision (a defined non-volatile storage region, kept out of the flashed program region), not a redesign, and removes a real, easy-to-miss failure mode (a user reflashing and forgetting to restore configuration).
  - **The RTC-calibration half of this decision largely dissolves with RV-3028-C7 (audit note).** The "reflashing wipes the crystal offset" gotcha is a property of an uncompensated crystal RTC that depends on a host-managed trim value — the situation Witty Pi is in, and the situation PCF85063A would have put PowerMOD in. **A TCXO part carries its compensation internally and does not rely on a calibration constant living in PowerMOD's flash**, so there is no offset for a reflash to lose. Some drift-trim capability still exists on the part itself; it simply isn't state this firmware owns and can destroy.
  - **The EEPROM-separation decision stands regardless, on its own merits** — it was never solely about RTC calibration. User configuration, the power-on state bit, the wake time and its commit-time validity flag, and the event log all need to survive a reflash, and all of them are PowerMOD's own state. This is now the load-bearing justification rather than the RTC offset.
  - **↪ PY32F003 (instock-parts branch) — the separation becomes a firmware/tooling requirement, not a free hardware property.** On the ATtiny, EEPROM was a physically distinct array that programming never touched, so "survives a reflash" was automatic. The PY32 has **no EEPROM**: this state lives in a reserved **flash** page, in the same array as the program. It survives an *application-only* update (the reserved page is left unerased), but a **full chip-erase reflash wipes it**. Requirement: (a) firmware reserves a dedicated flash page for config/log/state and locates it outside the application image; (b) the field-update procedure must be config-preserving (SWD application-region write, or the bootloader's app-only path) — **not** mass-erase — unless a config reset is intended. Documented for the operator in user guide §4.5. Endurance is not a concern (100 k cycles, batch-1 claim #8).
- **Mains/battery status — confirmed, exact register.** Register #7, "power mode": 0 = USB-C 5V (mains), 1 = LDO/battery (numbering appears to vary slightly by Witty Pi model/revision — confirm exact values against final firmware chosen as reference). Matches the mains/battery signal designed independently in Section 3.5.
- **Hysteresis/re-arm thresholds — real, concrete precedent found.** Register #19 ("low voltage threshold," multiple-of-10 encoding, default disabled, configurable 3.0–4.2V per the manual text) and register #22 ("recovery voltage threshold," same encoding, independently disableable) are **two distinct, separately-configurable registers** — direct, concrete precedent for the cut/re-arm split designed independently in Section 3.5. Use this as the reference encoding scheme rather than inventing a new one.
  - **Encoding adopted; the disabled-by-default value rejected** — see the fallback-safety section above for why the low-voltage threshold ships enabled instead.
  - **A third threshold was missing entirely: the critical-battery warning (audit finding, added).** The Battery LED promises a "critically low, approaching the emergency-cutoff floor" state — an early warning giving an operator a chance to intervene *before* the cutoff fires. **Nothing defined when it fires.** There was a cutoff threshold and a recovery threshold and no warning threshold, so the documented behaviour had no trigger. Same shape as the shutdown-pending flag: a state the guide instructs people to rely on, with no mechanism behind it. **Requirement: a `critical voltage threshold` register**, same volts×10 encoding as its two siblings, default **3.3V** (a warning window above the 3.0V cutoff), `0` to disable. **It belongs adjacent to the other two in the register map** — three thresholds describing the same voltage axis should be read together, and this document's recurring failure has been precisely that adjacent-but-different things get conflated when they're written apart.
  - **The pair is cross-validated at commit — a decision the register map forced (adopted).** If both thresholds are enabled, recovery must be **strictly greater** than cutoff; an inverted commit is rejected whole. This was initially left unvalidated on the grounds that it collided with the "board doesn't compensate for host mistakes" philosophy. **It doesn't: that principle is specific to wake-time writes**, where absolute and relative modes are both legitimate and the board cannot know which is intended (the whole point of the wording-correction section below). Configuration is already a validated category — values outside 3.0–4.2V are rejected today and nobody considers that a violation. An inverted pair is the same class of check, and unlike a wake time it has no valid reading: it is not an unusual configuration but a broken one, reintroducing the exact oscillation this split exists to prevent, at the moment the cell is least able to tolerate it. **"Independently configurable" is preserved in the sense that matters** — either threshold can still be set or disabled on its own. Strictly-greater is enforced; adequate margin (≥0.3V; the defaults give 0.6V) is documented advice rather than a mandate, matching the proportionate-response pattern used for the voltage jumper and the port-swap risk.
- **"Minimum off-duration / rapid-reboot protection" — the wrong register was adopted, and the gap stayed open for the entire review (found 2026-07-15).** Register #21, "delay before power cut," default 50 (5 seconds; later 7.0s per their changelog) — a deliberate, configurable pause between "shutdown requested" and "power actually cut." This was recorded as *"Resolves the Section 9 timing-domain gap directly."*
  - **It does not, and it never did. They are different mechanisms.** *Minimum off-duration* is **how long power stays OFF before it may come back on**. *Delay before power cut* is **how long before power is removed**. One governs the gap between off and on; the other governs the gap between request and off. **Adopting the second to close the first closed nothing** — and it passed unchallenged because the two names sit next to each other. Same failure as crediting the load switch to the charger, or the thermistor to a 2-pin connector: **a capability assigned to the wrong mechanism because the words were adjacent.**
  - **The gap is real and reachable by at least three paths.** A Linux SBC power-cycled too fast may not discharge its rails below the SoC's power-on-reset threshold, so it never cleanly resets — it hangs, or boots into a strange state. PowerMOD can currently do this via: **(1)** the watchdog, whose own specification said it "immediately re-powers the host" — a **zero-second** power cycle, written while believing this gap was closed; **(2)** a host that writes a wake time a few seconds out and then requests power-off (legal — no validation, by design); **(3)** button off followed immediately by button on.
  - **Requirement: a `minimum off-duration` register, distinct from both delay registers.** After power is cut, PowerMOD does not re-power the host until this duration has elapsed, **whatever the wake trigger** — scheduled, missed alarm, button, watchdog, max-sleep, or power restore. Default **5.0s**, enough for typical SBC rails to discharge; configurable, `0` to disable. **It is invisible in normal operation** (scheduled wakes are minutes or hours away) and only engages in exactly the pathological cases above, which is what a guard should do.
  - **The delay-before-cut register is still worth having on its own merits** — it is the only warning window a polling host gets before a button press or low-voltage cutoff (see below). It simply never addressed this gap.
  - **Split into two registers (audit finding).** A single delay value is being asked to serve two purposes with opposite optimal settings, and no one value satisfies both:
    - **Host-requested cut:** the host has already declared its shutdown complete. The delay accomplishes nothing except burning battery while a finished host sits powered. Wants to be **short**.
    - **Button-initiated or low-voltage cut:** the host has *not* declared anything and may be mid-work. This delay is the entire grace window it gets. 7 seconds is nowhere near a Linux host's realistic worst-case shutdown, so at the inherited default this window is close to decorative. Wants to be **long**, and sized to the actual host.
  - **Requirement: two independent registers — `delay before cut (host-requested)` and `delay before cut (forced)`** — so an integrator can set the forced delay to their host's real shutdown time without paying that cost on every ordinary scheduled sleep. Two bytes each.
  - **Host-requested default: 0, not Witty Pi's 7s (2026-07-15). The inherited number's justification does not transfer, and keeping it is measurably expensive.**
    - **Why the precedent doesn't apply.** Witty Pi's 5–7s exists because *their* architecture tells the Pi to shut down and then waits for it to finish — the delay **is** their shutdown window. PowerMOD's host issues `POWER_OFF_REQUEST` **only once its shutdown is already complete** (Section 2, the entire premise of the two-command design). There is nothing left to wait for. The number was inherited without checking whether the reason came with it.
    - **What it actually costs, measured.** A halted Raspberry Pi draws **~220–300mA depending on the model** (triangulated across Pi 3/4/5 — claims row 14) — it does *not* power itself off (verified: "The Raspberry Pi has never had a power management circuit that allows it to power off — what you call power off is actually HALT"). So the delay is not idle time, it is **220mA of drain on a host that has already finished working**: 0.43 mAh per cycle, which on a 15-minute schedule is **~41 mAh/day — about 2% of a 2000mAh cell, every day, for nothing.**
    - **The register is retained** (defaulting to 0) rather than deleted, so an integrator with a host-specific settle requirement can still ask for one. But nobody should pay for it by default.
    - **A related worry this does *not* address, because it is a different mechanism:** a host that "sees power and reboots" after finishing its shutdown. That is not Pi behaviour — a halted Pi stays halted until GPIO3 is grounded or power is cycled. The real version of that concern is **rapid re-power**, which needs the minimum-off-duration register below, not a pre-cut delay.
- **Does the low-voltage emergency cutoff go through the delay, or bypass it? — contradiction between documents, now resolved (audit finding).** The three documents disagreed outright. The state-machine diagram had `EMERGENCY_CUTOFF` bypassing `SHUTDOWN_PENDING` entirely with **no delay**, describing it as "the one transition that ignores the host completely." This spec's button decision and the user guide both routed the low-voltage cutoff **through** the same configurable delay as any other power-off. Not a wording difference — it changes the state machine.
  - **Resolved: the emergency cutoff routes through `SHUTDOWN_PENDING` and observes the forced delay, like every other power-off.** The "bypass" framing sounds safety-minded but buys nothing real: the cutoff floor (3.0–3.3V) sits well above the board's own minimum operating voltage, deliberately and with margin, so the floor is not a cliff that seconds matter at. The cost of the delay at the floor is negligible — a few seconds at ~1A from a typical cell is a fraction of a percent of its capacity — while the benefit is real, since the forced delay is the only warning window a polling host ever gets (Section on the button), and a battery-dying-under-you shutdown is exactly when a host would most want to flush state.
  - **The bypass also cost a second code path** for no behavioral gain. **All three triggers — host command, button press, low-voltage — now enter `SHUTDOWN_PENDING` identically**, and the recorded trigger reason selects only two things: which terminal state is entered on expiry (`OFF` vs. `EMERGENCY_CUTOFF`, the latter holding until the recovery threshold or mains returns), and whether a button press can cancel it (above). This makes the state machine smaller, not larger.
  - **Unchanged, and worth restating so this isn't misread as a softening:** the cutoff still requires zero host cooperation, still fires whether or not the host acknowledges anything, and still cannot be cancelled by any means. It is not a request. The delay is a fixed grace window before an unconditional action, not a negotiation.
- **Wake alarm firing while the host is already on — Witty Pi's caching approach considered, then rejected as not applicable.** Witty Pi caches a wake alarm that fires early and fires it after the next power-loss, because their design has a self-perpetuating ON/OFF schedule cycle where dropping an alarm would silently and permanently break the cycle (no "next" alarm gets scheduled). **PowerMOD's architecture doesn't have this problem**: there's no repeating cycle baked into the board — the host writes one wake time at a time, fresh, whenever it next goes to sleep. So a wake alarm firing while the host is already on doesn't orphan anything; the host will write a new wake time whenever it actually sleeps again. **Resolved: NOP.** A wake alarm arriving while the host is on does nothing — trying to "honor" it later (e.g., by force-cycling power after the host eventually sleeps) would mean PowerMOD inferring intent from a schedule already overtaken by events, which is inconsistent with the core "host owns the decision" design principle (Section 2). This also fully resolves the host-reboots-mid-sequence case from Section 3/9: whether the host reboots on its own or the wake alarm simply arrives during an already-running session, the outcome is the same harmless NOP, with no special-case logic required.
- **Physical manual button — Witty Pi's "graceful signal" approach considered, then rejected on further review.** Witty Pi's button, while host is on, signals a graceful shutdown request via a dedicated GPIO pin the host software actively watches. PowerMOD initially adopted this pattern, but it doesn't actually work given PowerMOD's core design: **PowerMOD has no signal/interrupt line to the host at all (I2C-only, host-initiated) — a "graceful shutdown request" would just be a register write nobody may ever read**, since nothing requires the host to be polling at that moment. Presenting this as a graceful request would be misleading. **Resolved instead as an honest blunt override**: button press while host is on immediately begins powering off (after the configurable *forced* delay — see the split above), with no wait for host acknowledgment — functionally closer to pulling the plug (with a short grace window) than to a polite request. Gets its own distinct reason code (button-forced-off, separate from scheduled shutdown and low-battery cutoff) so a host can distinguish the cause after the fact. This is a direct consequence of the I2C-only, no-signal-line design already chosen (Section 2) — worth noting as a real tradeoff of that choice, not a free win.
- **Hold-to-force — added 2026-07-15. The button had no way to cut power *now*.** As specified, every button-initiated power-off waited out the full forced delay (default 30s). That delay exists to give a *polling* host a warning window — but if the host is hung, or simply isn't polling (most won't be), the wait buys nothing and the operator standing at the device has no way to skip it. Every PC and phone in existence lets you hold the button to force power off; PowerMOD didn't.
  - **Requirement: holding the button for ≥5 seconds while the host is on cuts power immediately** — no delay, no cancellation, its own reason code. This is the genuine "pull the plug" escape, and it is the *only* path that bypasses the forced delay.
  - **It composes cleanly with tap-to-start rather than replacing it.** Pressing the button starts the forced delay as before; *continuing to hold* escalates that pending shutdown to an immediate cut at 5s. No press/release disambiguation is needed — a hold simply overtakes the delay it already started.
  - **Cancellation belongs to the tap path only, and that asymmetry is the point.** A tap starts a delay you can take back (below); a 5-second hold is already a deliberate act, and there is nothing to cancel because the cut is immediate. **The two paths divide by intent:** tap = "start shutting down, I might change my mind"; hold = "off, now."
  - Matches the factory-reset gesture's 5s hold (Section on recovery) for consistency — the two cannot be confused, since one happens while running and the other only during power-up.
- **Button press during an in-progress shutdown cancels it — but only a button-initiated one (audit finding, added).** As previously specified, an accidental button bump on a running host was an *unstoppable* ungraceful power cut: the operator can watch the LED indicate a pending shutdown and has no way to abort, even standing right next to the device. Given that the button is deliberately a blunt override with no host handshake, an escape hatch for the misfire case is worth having — and it is genuinely cheap, since the delay timer already exists and already has a recorded trigger reason.
  - **Scoped by trigger, deliberately.** Cancelling a **host-requested** cut would be actively harmful: the host has already completed its shutdown sequence and is not coming back: cancelling would leave PowerMOD powering a dead host indefinitely, with no one left to re-issue the command. Cancelling a **low-voltage forced** cut would defeat a safety floor whose entire purpose is to be non-negotiable.
  - **Requirement: a second button press during `SHUTDOWN_PENDING` aborts the shutdown and returns to ON, if and only if the pending shutdown was itself button-initiated.** Otherwise the press is ignored. Cancellation is recorded as its own event so the log doesn't imply a shutdown that never happened.
- **A real, genuinely new failure category Witty Pi's manual documents at length, and that this board's I2C-only design likely avoids by construction:** Witty Pi relies on multiple GPIO pins (button signal on GPIO-4, a "system is up" signal on GPIO-17, and passively monitoring the host's serial TXD pin to infer shutdown state) — and their own troubleshooting section (15.5) describes real, common failures from this: GPIO-4 conflicting with the host's 1-Wire interface (causing an instant shutdown-on-every-boot loop), serial port misconfiguration causing false "system is off" reads, and GPIO-4 not reaching a stable state fast enough during boot on weak power supplies. **Because this board's design is I2C-only** (no GPIO pin-sharing, no passive serial-pin monitoring — Section 2's whole point was avoiding inferred signals like this), **this entire category of failure is very likely avoided by construction, not just mitigated.** Worth stating explicitly in the spec as a real, structural advantage of the two-command I2C design, confirmed by seeing what actually goes wrong with the GPIO-based alternative in the field.
- **Reverse battery polarity — re-examined 2026-07-15. The precedent is real, the reassurance it offered was not, and the mitigation defends against the wrong failure.** Original finding: no active protection found in Witty Pi's design; their manual recommends soldering battery wires to dedicated pads for high-current setups; therefore silkscreen-only is "consistent with — and mildly reassuring precedent for" PowerMOD's approach.
  - **What the precedent actually is (verified against their manual).** Witty Pi 4 L3V7 has a **PH2.0 2-pin battery connector** — the same part as PowerMOD — *and* three big pads on the back. Their §11.1: "**If you expect high current consumption** from your device, it is recommended to solder the battery wires directly to 'BAT' and 'GND' pads. That will **minimize the contact resistor and also voltage drop** on the connector." **The pads exist for contact resistance at high current. They have nothing to do with polarity.** An earlier audit pass described them as "sidestepping the connector," which inferred a purpose from adjacency — the same error this document keeps recording elsewhere.
  - **The word "polarity" does not appear once in their 53-page manual.** So the precedent transfers exactly: same connector, no protection, **and no warning either**. "Consistent with Witty Pi" is true, and it means both boards carry the hazard unmitigated — not that it is mitigated.
  - **The real failure mode is not the one silkscreen addresses.** JST-PH is **keyed**, so a user physically cannot insert it backwards. But **JST-PH polarity is not standardised** — roughly half of cell vendors wire positive to one pin and half to the other (SparkFun's own documentation warns: "depending on the manufacturer they may be reversed"). **So the reversed cell plugs in perfectly.** Silkscreen tells the user which pin is positive on *our* board; it cannot tell them what their vendor did inside the plug. **The mitigation answers a question the keyed connector already prevented anyone from asking.**
  - **This collides directly with "works with any 3.7V-nominal Li-ion or LiPo battery."** The broader the battery support, the more certain that some user brings a cell from a vendor with the opposite convention. The promise manufactures the hazard.
  - **Decision: no protection circuitry; documented instead — but documented in terms the user can act on.** The prior guide text ("connect the battery according to the labeled polarity on the board") is **impossible to follow** and has been replaced. The only actionable step is metering the cell's own connector before it approaches the board, and that is now what the guide says, along with the labelled BAT/GND solder pads as an unambiguous alternative for anyone building more than a handful. **Reverse polarity on a LiPo can cause fire; if this decision is ever revisited, that is the reason.**

**Status 2026-07-15: two of these three are now closed by the part selections. Neither needed new design — the answers fell out of datasheets once real parts existed.**

- **Inrush current handling on host power-up — CLOSED. Handled in hardware by the TPS63020's soft-start; no design action needed.** Per its §7.4.1: after being enabled, "the average current limit ramps up from an initial 400 mA following the output voltage increasing… **the output voltage overshoot at start-up, as well as the inrush current, is kept at a minimum. The device ramps up the output voltage in a controlled manner even if a large capacitor is connected at the output.**" A host's bulk capacitance *is* that large output capacitor, which is precisely the inrush case this question was about.
  - **Bonus, unasked for: short-circuit protection.** "When the output voltage does not increase above 1.2 V, the device assumes a short circuit" and holds the current limit down. A shorted host connector is therefore bounded without any additional part.
  - **This is a *fourth* thing the 4-switch topology gives, and it is worth noticing the pattern.** The converter is what energises the host rail (via load disconnect), so **the converter's soft-start is what governs inrush.** Had the original boost-plus-load-switch design been built, the load switch would have connected a settled rail straight onto the host's bulk caps with no controlled ramp — inrush would have been a real problem needing a real answer. The buck-boost decision keeps quietly paying for itself: 3.3V support, load disconnect, inrush control, short-circuit protection.
- **PowerMOD's own thermal operating range — CLOSED, and the answer is not what the question assumed. The range is set by the *batteries*, not the silicon.** Every IC now selected is datasheet-rated **−40°C to +85°C** or wider, and they agree almost exactly:

  | Component | Operating range | Source |
  |---|---|---|
  | **TP4056 charger** | −40°C to +85°C | datasheet: "Operating Ambient Temperature Range: -40℃～85℃" — **identical to the MCP73871 it replaces, so the reversal does not move this bound** |
  | BM8563 RTC (was RV-3028) | −40°C to +85°C | datasheet — same range, bound unchanged |
  | TPS63020 buck-boost | −40°C to +85°C | "free air temperature range" |
  | ATtiny1617 | −40°C to +105°C | datasheet (wider; not binding — same die/range as the 1616 it replaces) |
  | JST PH connector | −40°C to +105°C | datasheet (wider; not binding) |
  | **CR2032 coin cell** | **−30°C to +60°C** | Energizer datasheet: "Operating Temp: -30C to 60C" |
  | LiPo (integrator's own) | discharge typically −20 to +60; **charge 0 to +45** | cell-dependent |

  - **Answer: −40°C to +85°C for the electronics; −30°C to +60°C for a board with the coin cell fitted; narrower still with a LiPo.** The silicon was never the constraint — it is uniformly the widest-rated thing on the board.
  - **The +60°C ceiling is a real concern for the flagship deployment, not a theoretical one.** An outdoor enclosure in direct sun readily exceeds 60°C, which is *the coin cell's limit*, not the electronics'. Worth being explicit: the part added for remote-solar resilience is also the part that most restricts where a remote-solar box can sit.
  - **Noted tradeoff, not a reopening:** a supercap is typically rated to the silicon's −40/+85 and would not narrow the range. The coin cell was chosen deliberately for multi-week outage coverage (see RTC backup power); this is a real cost of that choice, recorded so it isn't rediscovered later.
  - **The RTC-accuracy half of this question is separate and already answered elsewhere:** an earlier audit pass claimed the part's "integrated TCXO" handled it — **withdrawn, RV-3028-C7 has no TCXO.** Accuracy over temperature follows the crystal parabola and is now quantified and documented rather than compensated (see the RTC part decision). Witty Pi's onboard sensor remains the only temperature-compensation precedent in this space, and PowerMOD deliberately has no equivalent — though the option costs zero parts if ever wanted.
  - **This item previously hid a second, unrelated question, now separated (audit finding):** *battery charge-temperature qualification* — whether it is safe to charge the cell at the current temperature — is a different concern from the board's own operating range, and was silently absent from every document. **The separation stands and was worth making; the resulting requirement was then withdrawn** once it became clear the 2-pin battery connector gives the TS pin no route to the cell — see the charging-below-freezing section, where the hazard is now documented as the integrator's responsibility. **What remains open here is only the board's own range**, which is a component-selection question and not a safety one.
- **Component lifecycle/availability — still a project decision, but the evidence is now favourable rather than unknown.** Every selected part is a mainstream device from a major vendor, and each datasheet shows a long revision history — which is the best available proxy for continued supply:
  - **MCP73871** (Microchip): datasheet revised **2022**, originally 2008 — 14 years and still maintained.
  - **TPS63020** (TI): revised **2019**, originally 2010.
  - ~~RV-3028-C7 (Micro Crystal)~~ → **BM8563 (PCF8563-class)**: the architecture is a 25-year NXP mainstay with multiple clone vendors — lifecycle risk *improved* by the swap. **ATtiny1617** (Microchip): current, actively-marketed lines. The 1614/1616/1617 share one datasheet and one die family, so the 20→24-pin move carries no new lifecycle risk.
  - ~~**TP4056 is no longer in the design**, which incidentally removes the one part whose vendor documentation was thinnest…~~ — **REVERSED 2026-07-15, and this is the honest cost of that reversal.** TP4056 is back, and **every word of the original concern comes back with it**: a 3-page datasheet with no revision history, from a vendor with no Western distribution guarantee. The reversal is justified on architecture and cost, **not on this axis, where it is a genuine regression.**
    - **Mitigating, but only partly:** TP4056 is arguably the most-cloned lithium charger in existence — [multiple LCSC vendors](https://www.lcsc.com/product-detail/Battery-Management-ICs_TPOWER-TP4056_C382139.html) list it (TPOWER, UMW, TECH PUBLIC) with 60,975 in stock at one alone. **Ubiquity is a different property from vendor commitment**, and it cuts both ways: second-sourcing is trivial, but no vendor owes you a roadmap, and "clone" implies the parts are not guaranteed identical. **Qualify the specific vendor's part, not "a TP4056."**
    - **The OR-ing PFETs are jellybean parts** and raise no lifecycle question of their own.
  - **What this does *not* establish:** actual stock, pricing, or NRND status, none of which a datasheet can tell you. Check distributor lifecycle status at BOM time. The point of this note is only that nothing selected is exotic or obviously end-of-life.

**Resolved — I2C bus contention/stuck-bus recovery, not a gap requiring new design, already covered by existing safety design for a different reason.** Recovering a globally stuck/jammed I2C bus is inherently a host/bus-controller responsibility — a peripheral device on a jammed bus has no way to un-jam it from its own side, regardless of how it's designed. The worst-case outcome of total I2C unavailability (host can't schedule, can't issue "cut now") is already bounded by the independent, I2C-free battery-voltage safety floor (Section 4.3/1.1): that floor doesn't depend on I2C communication at all, and doesn't care *why* the host went silent — crashed, hung, or genuinely unreachable over I2C all end up protected the same way. No new design needed. The one real, narrower responsibility that remains: PowerMOD's own firmware should be built/tested not to itself cause a bus lockup (e.g., never hold a line low indefinitely on an internal firmware fault) — a standard firmware-quality concern for any I2C peripheral, not a special feature to design in.

## 9. Gaps surfaced by a user-walkthrough (normal usage + failure scenarios)

Walking through the device as an actual user (unbox → connect → set time → schedule → shutdown → wake → steady-state → battery degradation → on-site troubleshooting) surfaced gaps the requirements-driven review hadn't hit. Several were subsequently resolved by reviewing Witty Pi 4 L3V7's full user manual (Section 8) — noted below.

- **Status indicator (LED or similar) — resolved, revised to 4 states / bicolor Power LED.** Two LEDs:
  - **Battery LED — revised from 2 states to a fuller scheme, also bicolor.** Originally scoped as charging vs. charged/done only — this left the most common real operating condition (running on battery, no mains present) with no representation at all, indistinguishable from "no battery installed" (mains-only mode) since both showed as simply "off." **Resolved with the same color+blink approach as the Power LED**, using a bicolor LED (same low-cost/complexity reasoning as above):
    - Solid, color A = charging (mains present, battery below full).
    - Solid, color B = charged/full (mains present, battery at capacity).
    - Slow blink, color B = discharging normally (on battery, no mains, battery healthy) — gives the normal/common battery-powered operating state an actual signal instead of silence.
    - Fast blink, color B = battery critically low — **below the configurable critical voltage threshold** (default 3.3V), which sits above the cutoff floor. An early warning *before* the emergency cutoff triggers, giving an operator a chance to notice and intervene. **The threshold register was missing entirely until the audit** — the state was documented with nothing to fire it; see the critical-threshold requirement in the explicit-scope-decisions section.
    - **Rare flash (~10s), color B, below the cutoff floor — reassigned here from the Power LED (user review, 2026-07-15).** The battery's own LED now tells the whole discharge story monotonically: **slow → fast → rare flash → dark (depleted)**. The first fix (semantic review, item 2) made this LED go *dark* at the battery's worst moment and had the Power LED narrate a battery crisis — urgency inverting on one LED while the signal sat on the wrong one.
    - Off = no battery present (mains-only mode). Now unambiguous, since "off" no longer also has to cover the healthy-discharging case.
  - **Host power status — four states, not three (revised).** Originally three states (powering / sleeping / emergency cutoff) didn't distinguish "sleeping with a wake scheduled" from "sleeping with no wake scheduled" (e.g. after a button-forced-off with no pending schedule) — both looked identical, but mean materially different things to an operator (one will recover on its own, the other won't). **Resolved via a 3-lead bicolor LED (e.g. red/green) instead of a single-color one** — genuinely low additional cost/complexity: one extra GPIO drive line (simple digital output, no PWM or driver IC needed), a bicolor package is only modestly larger/costlier than single-color, well short of true RGB's added pin count and complexity (RGB needs 3 independently-controlled channels for a "millions of colors" capability this use case never needed). Encoding: color + blink pattern together, redundantly, so states are distinguishable even if a viewer can't reliably judge blink rate alone:
    - Solid, color A = powering host.
    - Slow blink, color A = sleeping, wake scheduled.
    - Slow blink, color B = sleeping, no wake scheduled.
    - **Dark during emergency cutoff — reassigned 2026-07-15; the Battery LED carries the rare flash.** A dark Power LED reads as exactly what is true: the host has no power. Dark Power + flashing Battery = emergency; all dark = depleted. (Originally a fast blink, then a rare flash on this LED — see the emergency-cutoff indication decision below: that state runs on a cell already below its protection floor, so it is the one state where the indicator must be as cheap as possible.
  - **Stated limitation, not a gap:** below the board's own minimum operating voltage, it can't stay alive to drive any LED — "no LED activity for an extended period" is itself a signal (likely fully dead battery), and the user guide should state this explicitly.

### LED electrical specification — the scheme above specified everything except what it costs (audit finding, 2026-07-15)

**The scheme specifies colour and blink *rate*. Drain is determined by drive current and *duty cycle*, and neither was specified anywhere.** So the behaviour was fully documented and its power cost entirely undefined — in a design whose Scope table carries a **~0.3mA quiescent budget** as though it were settled.

**Costing it reveals the LEDs are the dominant term, and nothing else is close.** At a typical 2mA indicator drive:

| Condition | Naive implementation | Share of the 0.3mA budget |
|---|---|---|
| Host off, on battery — **both** LEDs slow-blinking (Power: "sleeping, wake armed"; Battery: "discharging normally") | ~0.20 mA | **~67%** |
| **Emergency cutoff — both fast-blinking** | ~1.0 mA | **~330%** |

For comparison, the parts checked against this same budget elsewhere in this document: the buck-boost's quiescent is **8%** of it, the RTC's backup current **0.01%**. **The budget was being validated against terms that are noise, while its largest consumer went uncosted.**

**Resolved, and cheaply: rate and duty are independent.** A 20ms pulse every 250ms *looks* like a fast blink and draws 8%, not 50%. The scheme's visual language survives untouched; it just needed its power specification written down. Smoke detectors have made this trade for decades — a ~10ms flash is unmistakable.

**Requirement: LED drive ~2mA, with duty cycle specified per state, not just rate:**

| State | Pattern | Duty | Avg/LED | Notes |
|---|---|---|---|---|
| Solid | continuous | 100% | 2 mA | **Only ever occurs on mains, or with the host running.** Charging/charged both require mains present; "powering host" means the host is drawing ~1A, against which 2mA is noise. Solid is never a quiescent concern. |
| Slow blink | 20ms / 2s | 1% | 0.02 mA | Both LEDs together: ~13% of budget. |
| Fast blink (critical) | 20ms / 250ms | 8% | 0.16 mA | Visually urgent, and the state is inherently short — it is the last stretch before cutoff. |
| **Emergency cutoff** | **20ms / 10s** | **0.2%** | **0.004 mA** | **Revised — see below.** |

Final values tune on hardware for visibility; the requirement is that **duty cycle is specified and budgeted**, not left to whatever the firmware author picks.

### Emergency-cutoff indication — revised from fast blink to a rare flash. The old behaviour contradicted the cutoff's purpose.

**The low-voltage cutoff exists to stop the cell being over-discharged. It then fast-blinked two LEDs at ~3.3× the entire quiescent budget, on a cell already below its protection floor.**

**The drain was inversely correlated with the battery's ability to sustain it** — which is exactly backwards:

| State | Cell condition | Old LED drain |
|---|---|---|
| Charging / host on | mains present; drain irrelevant | highest (solid) |
| Sleeping, healthy | on battery, plenty left | 0.1 mA |
| Critically low | nearly empty | **raised to 0.5 mA** |
| Emergency cutoff | **below the floor** | **highest at 1.0 mA** |

Both "get attention" states fire when the cell has least to give. On the ~100mAh tail below the 3.0V cutoff, this cuts survival from **~14 days to ~3** — so **the cutoff did not stop the over-discharge it exists to prevent; it slowed it, while the indicator hurried it along.**

**Resolved: after an emergency cutoff, a single 20ms flash every 10 seconds (colour B).** Rationale:
- **It preserves the only distinction that matters.** An operator must be able to tell "cut off, waiting to recover" from "completely dead" — and this design already declares that *total* LED silence means a flat battery. A rare flash keeps those apart at ~0.2% duty.
- **Going fully dark was considered and rejected** for exactly that reason: it would collide with the depleted-battery signal.
- **It is still visible.** A brief flash on a dark board draws the eye; this is the one state where slow-and-rare is the *right* signal anyway — nothing is going to change quickly.

**Also worth stating plainly, since the audit surfaced it:** PowerMOD's own quiescent draw continues after the cutoff and keeps discharging the cell below its floor. The floor slows over-discharge; it cannot stop it while the board stays alive to detect recovery. Minimising post-cutoff draw is therefore a real design goal, and the LED was the largest and most easily removed part of it.
- **Minimum off-duration / rapid-reboot protection — resolved, see Section 8.** Adopt a configurable "delay before power cut" register, directly precedented by Witty Pi's register #21.
- **Physical manual on/off control — resolved, see Section 8 (revised).** Button present; power-on is unconditional (no host cooperation needed). Power-off is a blunt, immediate override with a grace delay — not a graceful request — because PowerMOD's I2C-only design has no way to notify a host of the button press if it isn't actively polling. Distinct reason code from scheduled/low-battery cutoffs.
- **Battery connector polarity protection — resolved, see Section 8.** Silkscreen/labeling only, consistent with Witty Pi's own apparent approach (no active protection found in their design either) and the port-swap decision already made in the Interface section.
- **No debug/recovery path if I2C itself is the problem — resolved, see Section 8.** Not a gap requiring new design: the independent battery-voltage safety floor already bounds the worst case regardless of cause (crashed host, hung host, or genuinely unreachable I2C all end the same way). PowerMOD's own responsibility is limited to not causing a bus lockup itself — normal firmware quality practice, not a special feature.

## State-machine details (second review pass) — resolved

A further review pass surfaced smaller, more subtle state-machine questions — the kind that only show up when mentally simulating transitions rather than reading requirements top-down. Resolved below; most are simple, low-cost definitions rather than design changes.

- **Writing a new wake time while one already exists.** Resolved: PowerMOD stores exactly one wake time. A new write always replaces the previous one — no queueing, no "earliest wins" logic, no rejection. Simple and predictable.
- **Canceling a scheduled wake — genuine gap, now resolved.** No explicit "clear the wake time" operation existed. **Added: a clear-wake command/register value**, separate from writing a new (still-future) wake time, for a host that wants to cancel a pending schedule without immediately providing a replacement.
- **Partial/interrupted I2C write (power loss mid-transaction).** Real gap — a multi-byte wake-time write interrupted by power loss could leave a corrupted, partially-written value. **Resolved: wake-time writes use a shadow-register-and-commit pattern** — the host writes the full multi-byte value to a staging area, then a separate final "commit" write atomically applies it. An interrupted write leaves the previously-committed wake time untouched rather than a corrupted hybrid value.
  - **Read atomicity was never specified — added (register-map finding).** This document defines *write* atomicity carefully and says nothing about reads, while exposing a live-incrementing 32-bit counter. A sequential read of the RTC can tear: read the low bytes at `0x0FFFFFFF`, the counter rolls, read the high bytes, and the host gets a timestamp off by 256 seconds — plausible, wrong, and silent. **Resolved: reading the first byte of any multi-byte field latches a coherent snapshot of that field**, released when the pointer leaves it. The read-side mirror of the write protection already required, and the same class of bug.
  - **The commit step now also carries the past/future evaluation (audit finding).** At the moment of commit — and only then — the timestamp is compared against the RTC and flagged as either armed or already-expired. That flag is stored in EEPROM alongside the wake time and is what resolves the missed-wake vs. past-wake contradiction (Section on fallback safety). No new mechanism: the commit point already existed, it now decides one more bit.
- **Reason-code retention.** Resolved: **each of the two reason registers** (power-on and power-off — see the split in the explicit-scope-decisions section) is retained until overwritten by the next event **of its own kind** — not cleared on read, not time-limited. **Revised by the audit:** this rule previously named a single register overwritten by *any* event including scheduled wakes, which is precisely what destroyed the power-off reason during boot. Retention is per-register and per-category; a wake never clobbers a power-off reason.
- **`SHUTDOWN_PENDING` was not observable over I2C — genuine gap, now resolved (audit finding).** The user guide instructs hosts to "poll PowerMOD's status/reason-code registers frequently enough to detect that a button-initiated power-off is pending" — but no such register existed anywhere in this spec. The reason-code registers are by definition written *after* an event, so there was nothing to poll, and the guide documented a capability the design didn't have. Without it the forced-delay grace window is unusable by construction: a host cannot react to a window it cannot see. **Resolved: expose a shutdown-pending status flag** indicating that the delay timer is running and, via the trigger reason, why. One bit, readable over the existing interface.
- **Battery removed while running on battery (no mains).** Resolved, stated explicitly rather than left implicit: immediate power loss, same as unplugging any battery-powered device — no special handling, this is physically unavoidable.
- **Race: wake alarm reached during the delay-before-cut window.** Timeline: host issues "cut now," the configurable delay begins, and the scheduled wake alarm's target time is reached before the delay elapses. **Resolved: the in-progress shutdown wins.** The wake alarm is not acted on because the host isn't actually off yet — consistent with the existing "wake alarm while host is on is a NOP" resolution (Section on scheduled wake behavior), since from PowerMOD's perspective the host is still considered on until the delay elapses and power is actually cut.
- **Configuration/EEPROM corruption.** A new failure mode introduced by adopting EEPROM-persistent configuration (see Firmware Updates, above) — worth being honest that this wasn't a risk before persistent config existed. **Resolved: configuration is stored with a checksum/CRC; on a checksum mismatch at boot, PowerMOD falls back to factory-default configuration** rather than operating on corrupted values or refusing to boot.
  - **The fallback must be reported, not silent (audit finding, added).** As originally resolved, this said nothing about telling anyone. A silent revert is its own failure: an integrator who deliberately raised the low-voltage threshold, or enabled the watchdog, is now running a board that quietly did neither — with no symptom until the moment the setting was supposed to matter, which on an unattended deployment is exactly when nobody is watching. **Requirement: a readable status flag indicating that configuration failed its integrity check and factory defaults are in use** (`STATUS.CONFIG_DEFAULTED`, register map §3), recommended reading on every boot so a host can re-apply its configuration. One bit, and it converts a silent reversion into a detectable, correctable event.
- **Forgotten I2C address / general recovery path — genuine gap, now resolved.** No path previously existed for a user who changed the I2C address and forgot the new value. **Resolved: holding the physical button during power-up triggers a factory reset** (all configuration, including I2C address, reverts to defaults). This single mechanism covers both the "forgot my address" case and general "I've misconfigured this and want to start over" support scenarios — no need for two separate recovery gestures.
  - **A second, software factory-reset path exists over I2C — surfaced by the reverse audit, adopted with justification (was invented by the register map without spec backing).** The register map defines a command value that restores configuration defaults over I2C, which this document never asked for and which sits awkwardly against the sentence above ("no need for two separate recovery gestures"). Reconciled rather than removed:
    - **That sentence was about the forgot-my-address case, and the software path cannot serve it** — a host that doesn't know the address can't send a command to any address. The button gesture remains the only recovery for that, and is not duplicated.
    - **The software path serves a different purpose the button can't: a version-proof reset to a known baseline.** A host *can* already reset configuration by writing each register's default individually — so this adds no capability, only robustness. But "write every default" is brittle exactly where this document says drift is real: PowerMOD is explicitly meant to work "across as many hosts as possible… over a long lifetime," so a v1 host performing a register-by-register reset against v2 firmware would silently miss any config register v2 added. One command is immune to that.
    - **Kept on those grounds**, gated behind a distinct magic value, and with the same scope as the button gesture (configuration only — not the event log, not a pending wake time). If the version-drift argument is judged not worth a second mechanism, this is the one to cut: it is convenience plus future-proofing, not capability.
  - **Two under-specified details, now required (audit finding).** As written, this gesture collides with the button's ordinary power-on function and gives the user nothing to confirm against — hold the button, connect power, release, and there is no way to know whether a factory reset actually happened or whether the host simply booted. A recovery mechanism that can't be confirmed isn't much of a recovery mechanism, particularly for someone already troubleshooting. **Requirement — now satisfied with committed defaults (2026-07-15): hold = 5s (matching the force-off hold, so every deliberate hold gesture is one number), acknowledgment = both LEDs alternating for 2s, a pattern no other state uses; bench-check the feel only.** **Also pinned on user review: "power first connected" means cold boot from completely dark — no mains *and* no battery.** On a battery-backed unit the gesture is therefore rare by construction, and that is the feature: **a live system cannot be fat-fingered into a wipe** (a 5s hold on a running board force-cuts, a recoverable act). The full power-down it demands is the same one an I2C address change already requires — two recovery operations, one physical procedure. As originally stated: **(1) define an explicit minimum hold duration** to disambiguate the gesture from a normal press, and **(2) acknowledge the reset with a distinct, unmistakable LED pattern** using the existing bicolor LEDs — no new hardware, purely a firmware and documentation task. **Scope, explicitly defined (external review feedback adopted — previously unspecified): factory reset restores configuration registers only** (voltage thresholds, power-on mode, I2C address, delay-before-cut, etc.). **It does not clear the event log or a pending wake time.** Reasoning: the event log's entire purpose is durability through exactly this kind of disruptive event — clearing it at the moment someone is troubleshooting or recovering from a problem would delete the evidence of what led to needing a reset. A pending wake time is operational state, not configuration, and has no natural reason to be tied to a configuration reset.
- **Configuration write atomicity — resolved, extends the existing wake-time write protection uniformly.** Wake-time writes already use a staging-then-commit pattern to survive power loss mid-write (Section on state-machine details, above). **The same protection now applies to configuration writes** (voltage thresholds, power-on mode, etc.) rather than being a separate, undefined case — a config value corrupted by power loss mid-write is exactly as bad as a corrupted wake-time value, and applying one consistent write-safety model across all persisted register types is simpler to implement and reason about than maintaining two different ones.
- **Coin cell depletion.** Not previously addressed as a distinct case. Resolved: functionally identical to the existing first-power-up behavior (RTC resets to epoch) — a dead coin cell just means this can now also happen after the fact, not only at first setup. No new design needed; worth stating explicitly so it isn't mistaken for a new failure mode requiring separate handling. **Corrected by the audit:** this previously said "and the pending schedule is lost," which is no longer true — the wake time lives in EEPROM and survives a dead coin cell (Section on RTC backup power). What is lost is only the clock, which in turn makes any *absolute* pending wake time meaningless even though the value itself is still there; the RTC validity flag is how a host detects this.
- **Maximum-sleep safety timeout vs. a normally scheduled wake — which wins if both are set.** Resolved: whichever event's time arrives first. No special precedence logic needed beyond "check both, act on the earlier one" — the safety timeout is a ceiling, not an override.
- **Battery sags below cutoff immediately after boot (power restored, host begins booting, battery voltage dips under load right at the recovery threshold).** Already covered by the existing cutoff/recovery hysteresis (Section on fallback safety) — the recovery threshold is deliberately set with margin above the cutoff floor specifically to avoid this oscillation. No new design needed; confirmed rather than newly resolved.
- **Two I2C masters attempting to control PowerMOD simultaneously.** Not a supported configuration, consistent with the existing "one PowerMOD per host" scope decision (Explicit scope decisions, above) — extending the same reasoning to "one controlling host" as well. Not designed for; behavior in this configuration is undefined and unsupported.
- **A wake time being written while PowerMOD is already asleep, from some device other than the sleeping host.** Confirmed impossible under normal operation — a sleeping host has no power to write anything, and PowerMOD has no other I2C-connected device by design (single-host scope, above). No action needed.

## State-machine details (third review pass) — resolved

- **Simultaneous wake triggers — genuine gap, now resolved.** Multiple things can now cause a wake (button, RTC scheduled alarm, mains restore under Always-ON/Restore-previous-state modes, the safety timeout, battery recovering above the hysteresis threshold, a missed alarm firing on power restoration). No rule previously existed for two of these becoming true at effectively the same instant (e.g., the safety timeout expiring in the same second as a scheduled wake). **Resolved: PowerMOD performs a single wake regardless of how many trigger conditions are simultaneously true, and the reported wake reason follows a fixed priority order** (most-specific/most-informative reason wins — e.g., a scheduled wake takes priority over the safety timeout if both would fire together, since the scheduled wake is the more meaningful explanation). **Ordering now finalized in the register map (§8.3): watchdog recovery > button > scheduled wake > missed wake > max-sleep timeout > power restored** — most-fault-bearing wins, so a watchdog recovery is never lost to a coincident routine event, and the least-specific reason ("there is power now") ranks last precisely because it is almost always true alongside something better.
- **RTC validity flag after total power loss — genuine gap, resolved, and the conditional is now closed (audit update).** Previously resolved that a dead coin cell + dead LiPo + no mains results in the same behavior as first power-up (RTC resets to epoch, schedule lost) — but nothing let the host *detect* that this happened versus reading a plausible-but-arbitrary timestamp with full confidence. **Resolved: the RTC's hardware power-loss-detected flag is exposed as a readable I2C register.** Lets a host distinguish "RTC time is trustworthy" from "RTC has been reset since I last set it, don't rely on this for absolute scheduling" — directly useful for hosts using the absolute wall-clock time mode (Section 4.1).
  - **Now genuinely verified against the datasheet (2026-07-15), after a round-trip through an overclaim.** An earlier audit pass asserted this was "confirmed against a real part" when it had only been recalled; that was withdrawn and the original hedge reinstated. **It has since been checked properly: RV-3028-C7 Status register `0Eh`, bit 0 = `PORF` (Power On Reset Flag), cleared by writing 0 to the bit.** The requirement stands and the mechanism exists. Note the sequence — asserted, withdrawn, then actually verified — because only the third step is worth anything, and the first two were indistinguishable from it in the document until someone looked.
- **Configuration write verification — resolved, refines the existing read-time checksum decision.** Previously resolved: a checksum/CRC on configuration detects corruption at boot and falls back to defaults. **Added: configuration writes are also verified immediately after programming** (read back and compare, or equivalent), not just checked passively at next boot — catches a failed EEPROM/flash write at the moment it happens rather than only on the next power cycle. Both mechanisms are complementary, not redundant: write-verification catches an immediate write failure; the boot-time checksum remains the safety net for corruption from any other cause (e.g., flash wear, a botched firmware update).
- **Wake-time write while host is already running — confirmed, stated explicitly.** Already implied by the "new wake-time write always replaces the previous one" resolution (no distinction was ever made based on host power state at write time), but worth stating outright to remove any doubt: yes, this is legal, and behaves identically whether the host is currently on or off — it simply updates the stored wake time.
- **Wake time was write-only — read-back added (audit finding).** Only "write wake time" and "clear wake time" were ever specified; nothing let a host ask what wake time is currently stored, or whether one is pending at all. The value is already stored, so exposing it costs nothing beyond the register addresses. It closes a real gap for a host that rebooted unexpectedly mid-sequence (a case this spec explicitly supports) and wants to know whether its earlier write actually committed before deciding whether to write a new one. **Requirement: the committed wake time and its armed/expired flag are readable.**
- **Event-log timestamps are only as meaningful as the RTC (audit note, documentation-only).** For a host using relative-time scheduling and never syncing wall-clock time — a fully supported mode (Section on RTC default state) — every event-log timestamp is an offset from an arbitrary epoch, and restarts from epoch after any RTC reset. The log remains useful for *ordering* and for relative intervals ("three low-voltage cutoffs within an hour of each other"), which is most of its diagnostic value, but it cannot be read as wall-clock history. Not a design flaw and no action needed beyond stating it in user-facing documentation, so the log isn't mistaken for something it can't be in that mode.

## Wording correction: "no validation" vs. past-timestamp handling

External review correctly flagged an internal contradiction in how the past-wake-time behavior was worded: stating "PowerMOD does not validate wake-time writes" immediately followed by a rule that evaluates a timestamp against current time reads as a validation rule, even though it isn't a *rejecting* one. **Corrected wording (adopted as written by reviewer): "PowerMOD accepts any well-formed timestamp without rejecting it. If that timestamp is already in the past when evaluated, it is simply treated as expired and never triggers a wake."** This distinguishes *accepting the write* (never rejected, regardless of value) from *acting on the stored value* (which naturally depends on what that value turns out to be) — the underlying behavior is unchanged from the prior resolution, only the description of it is corrected.

## Telemetry scope — what gets exposed over I2C, resolved

Prompted by comparing against Witty Pi's real register map (input voltage, output voltage, output current, battery voltage all exposed). Rather than mirror their full set, each candidate was checked against a single rule consistent with the minimal-BOM philosophy: **expose it only if the hardware is already measuring it for PowerMOD's own operation — don't add new sensing hardware purely for telemetry.**

- **Battery voltage — expose (already have this).** Load-bearing for the hysteresis/cutoff logic itself (Section on fallback safety), not just telemetry — exposing the reading is free.
- **Input (mains) voltage — was "expose, pending confirmation"; confirmation failed, now dropped (audit finding).** The original reasoning: PowerMOD already needs to distinguish mains-present from battery-only for the status flag, and "the expectation is this is implemented as an ADC read… in which case exposing the actual voltage rather than collapsing it to a boolean is free." That bullet also said to **confirm rather than assume**, and named the exact failure condition — "if presence-detection is implemented some other way… this would need new hardware and should be dropped under the same rule." **It was checked. It fails.**
  - **Mains presence is a digital read, not an ADC read — since confirmed against the datasheet.** The MCP73871 now adopted (Section on charger and power path) performs input detection *internally* — that is how it decides to switch the power path — and exposes the result on its digital **`PG`** output. Mains-vs-battery costs one GPIO. **There is no input ADC for a voltage reading to ride along on.** (BQ24074's equivalent is `PGOOD`, defined as "pulls to VSS when a valid input source is detected" — the same digital arrangement, so this conclusion does not depend on the part choice.)
  - **So exposing input voltage requires new hardware** — a divider on the input rail plus an ADC channel — **purely for telemetry. That is verbatim the reason output voltage is dropped below**, which left this document briefly inconsistent: two telemetry candidates with identical hardware cost, resolved opposite ways, separated only by an assumption.
  - **Note the irony, because it is instructive: adopting the power-path IC is what invalidated this.** Under the original bare-TP4056 design with discrete mains detection, an input divider + ADC was genuinely plausible, which is what made "free" sound reasonable. A better charger removed the very hardware the telemetry was justified by. **A justification can be killed by an unrelated improvement**, and nothing re-checks it automatically.
  - **Also worth weighing: output voltage has the stronger value case of the two** (confirming the 3.3V/5V jumper — guarding a documented way to destroy a host — versus "is my supply delivering?"). If the stronger case was dropped for needing a divider and an ADC channel, the weaker one cannot survive on identical cost.
  - > **REOPENED 2026-07-15 by the charger reversal — and this document called the shot.** The bullet directly above reads: *"adopting the power-path IC is what invalidated this. Under the original bare-TP4056 design with discrete mains detection, an input divider + ADC was genuinely plausible, which is what made 'free' sound reasonable. A better charger removed the very hardware the telemetry was justified by. **A justification can be killed by an unrelated improvement, and nothing re-checks it automatically.**"*
    >
    > **The improvement has now been un-adopted, so the justification returns intact.** There is no `PG` pin any more. Mains presence is detected by **a divider on VBUS** — precisely the "new hardware" whose absence killed this feature. Put that divider on an ADC-capable pin (free on the ATtiny1617, same pin, same resistors) and **input-voltage telemetry is genuinely free again**, on exactly the reasoning originally offered.
    >
    > **What this demonstrates is worth more than the feature.** The document predicted that a justification could be silently invalidated by an unrelated change, and wrote it down. It did not predict that the *reverse* would also happen — that undoing the change would silently *re-validate* a dropped feature. **Nothing re-checks that automatically either.** Both directions need the same discipline: **when a part changes, re-run every decision that cited that part as its reason — including the ones the part caused you to say *no* to.**
    >
    > ~~Not adopted here~~ — **ADOPTED 2026-07-15, in a dedicated metrics review well after the reversal settled** (which was the deferral's whole condition). `INPUT_MILLIVOLTS` lives at the reserved `0x32`–`0x33`. The same review adopted `BOARD_TEMPERATURE` (the ATtiny die sensor + `SIGROW.TEMPSENSE0/1` factory cal — datasheet-verified, claims row 45) and `STATUS.CHARGE_COMPLETE` (the `STDBY` pin already driving the LED). **Output voltage was re-examined per this section's own rule and stays dropped — it still needs hardware.** Original deferral: **But it is now cheap rather than impossible, and the honest note is that output-voltage telemetry — dropped alongside it for the same cost reason, and explicitly judged the *stronger* case — should be reconsidered at the same time.** The old text's own rule applies: *"Reopening it should reopen output voltage too."*

  - **Dropped ~~(final)~~ — now reopened, see above.** `STATUS.MAINS_PRESENT`, ~~sourced from the charger's power-good output~~ **now sourced from a VBUS divider**, remains the supported way to ask the presence question — and is arguably *better* than a voltage threshold, since the charger asserts it based on whether the input is actually usable rather than on a number this design would have had to invent. **If input-voltage telemetry is ever wanted for supply diagnostics on solar deployments, that is a legitimate new requirement to argue on its own merits — a deliberate exception to this rule costing a divider, an ADC channel, and possibly a larger MCU — not a free win. Reopening it should reopen output voltage too.**
- **Output voltage — considered, dropped.** Would have real value (confirming the voltage-select jumper, Section on Extensibility, is actually set correctly, rather than only knowable by visual inspection) — but PowerMOD has no existing reason to *measure* its own output (it sets the output voltage via the buck-boost's feedback/select mechanism, it doesn't sense it). Adding this would require new hardware (an output-side voltage divider + ADC channel) purely for telemetry. Fails the "already free" rule — dropped in favor of minimal BOM.
- **Output current — considered, dropped.** Same reasoning as Witty Pi's own implementation shows this isn't free — requires a dedicated current-sense resistor, an additional ADC channel, and calibration/adjustment registers (visible in their real register map as a whole separate correction mechanism). No existing PowerMOD function needs this measurement. Dropped.

## Event log — reconsidered again, decision reversed, adopted

External review raised the absence of any event history (beyond the single most-recent reason code) independently across three separate review passes, framing it as "when someone asks why did my remote station reboot, they can just read the log." This was twice held as a deliberate non-goal on cost/complexity/write-wear grounds — but those costs were never actually quantified, only assumed to be significant enough to outweigh a repeatedly-requested feature. Worth actually doing the math before deciding a third time.

**The real cost, quantified:**
- One log entry = reason code (1 byte, reuses the existing enum) + timestamp (4 bytes, same format as the existing wake-time registers) = **5 bytes/entry**.
- 8 entries (the number every review independently proposed) = **40 bytes of non-volatile storage total.** Even a more generous 16 entries is 80 bytes — genuinely small relative to typical ATtiny-class EEPROM capacity, and this reuses the non-volatile storage mechanism already built for persistent configuration (Firmware Updates section, above) rather than requiring new infrastructure.
- **Write-wear, the one real concern, has a standard, well-known fix:** a circular buffer with wear-leveling across the log's allocated slots (write entry N to `address = (N mod 8) × 5`, rotating through all 8 slots rather than always overwriting the same bytes) spreads writes across the whole log region instead of concentrating them on one cell, multiplying effective lifespan by the slot count. This is standard, low-complexity firmware practice, not novel engineering.

**Decision: adopt.** Genuinely cheap once quantified (tens of bytes, a small, well-understood circular-buffer implementation), reuses existing non-volatile storage infrastructure, and resolves the single most independently-requested enhancement across the entire review process. **Specification: 8-entry circular log (reason code + timestamp per entry), wear-leveled via round-robin slot writes, readable by the host over I2C.** This supersedes the earlier "no event log" resolution (Section 9) and the prior two-round deferral — the underlying reasoning for those (added complexity, write-wear) turned out to be smaller in practice than assumed, and repeated independent demand was the signal that prompted actually quantifying it rather than continuing to estimate.

Two independent external reviews of this spec both suggested adding a dedicated signal/interrupt pin so the button (and potentially other events) could request a graceful shutdown from a cooperative host, with the existing blunt-override behavior kept only as a fallback. **Not adopted — this is the same design we considered and rejected earlier in this process (see the button behavior resolution above), and neither review's version resolves the reason it was rejected.** A signal pin only helps if the host happens to be polling or listening at that exact moment — nothing in PowerMOD's design guarantees that, and one of the reviews' own proposed implementations explicitly keeps a timer as a fallback specifically because the signal isn't guaranteed to be seen. That's not a fix for the underlying problem, it's an acknowledgment that the problem remains and needs the same fallback we already have. Adding a signal pin would reintroduce real complexity (a GPIO the host must actively poll or interrupt-handle, host-specific wiring/pin assumptions PowerMOD has otherwise deliberately avoided — see Section 2) for a benefit that only materializes when the host happens to be listening, which is precisely the condition the blunt-override design was chosen to not depend on.

**Audit note — this rejection stands, and two changes elsewhere sharpen rather than weaken it.** The shutdown-pending status flag now added (Section on state-machine details) gives a cooperative host exactly what the reviews wanted *without* a pin: a host that chooses to poll can see a pending power-off and react within the forced delay. That is the same benefit the signal pin offered, over the interface PowerMOD already has, with no new wiring and no host-specific pin assumptions — which rather makes the point that the pin was never what was needed. Separately, the opt-in watchdog (Section on fallback safety) addresses the *underlying* worry these reviews kept circling — a host that stops cooperating — from the opposite direction, without PowerMOD needing to signal the host at all.

One review also suggested rejecting wake-time writes that are more than ~60 seconds in the past, as a defensive validation measure. **Not adopted — this would break the relative-time scheduling mode already documented (Section on Time/scheduling)**, where a host deliberately never syncs wall-clock time to PowerMOD's RTC and instead always computes wake times relative to PowerMOD's own current (possibly still-epoch) clock value. A fixed "reject if more than 60 seconds in the past" rule has no meaningful definition of "past" in that mode, and would misfire for a legitimate, already-designed-for use case. The existing resolution (no board-side validation, host-side responsibility, relative-time as the recommended pattern for avoiding this exact class of bug) remains the better fit given both time modes this design explicitly supports.

## Appendix: unverified claims register — read before the BOM

**Why this exists.** The audit that produced most of the 2026-07-15 revisions above made the same mistake three times: **it named a component, costed it, and never traced what it physically connects to.** Each failure was caught only when someone asked "but how does that actually work?":

- **Temperature-qualified charging** — adopted as "10¢, the TS pin is already paid for," withdrawn once it was noticed the 2-pin battery connector gives the pin no route to the cell.
- **Input-voltage telemetry** — kept as "free, it reuses the mains-detect ADC," dropped once it was traced and the then-selected charger turned out to report mains presence *digitally*. (Later half-reopened when the charger reversal made mains detect an ADC divider after all — the drop was right *and* its reason didn't survive the year's shortest-lived part selection.)
- **The power-path IC's "four findings"** — one of the four was the TS pin, so the count was wrong the moment the first item fell.

**Status: three batches, all checked — batch 1: 2 of 8 false as written; batch 2: 0 false, 2 left open (1 since closed by the LDO selection, 1 closes only by measuring a real Pi); batch 3: 16 of 23 false; batch 4 (the RTC gates): 0 false, 5 confirmed — the deliberately-open vendor row closed when Belling's datasheet surfaced via an M5Stack mirror, and batch 2's Pi-halt row closed by triangulation the same day. The fact that each closed batch spawned the next is the most useful thing in this appendix** — and the worsening ratio tracks the claims getting closer to the BOM.

> **These tallies were themselves wrong until the self-description audit recounted them against the rows (2026-07-15).** The running batch-3 count had drifted 2 low ("12 of 21" against 14 ❌ rows), and batch 2's two *open* items had been counted as *false* — errors that happened to cancel, so the 16-of-37 total was right while both terms were wrong. **A tally that is never recomputed from its rows is just another self-description**, and it rots like the rest of them.

> **This register was declared closed, and within the same day's work it accumulated eleven new unverified claims.** Not through carelessness — through exactly the mechanism it was created to catch. Design work *generates* claims: every decision reaches for a fact about a part, and unless a discipline forces the question "did I read that or recall it?", the answer defaults to recall and looks identical on the page. **A claims register is not a task to complete. It is a standing tax on making decisions**, and the moment it reads "closed" it has stopped working.
>
> **The rule that follows: any decision that rests on a part doing something specific adds a row here, at the time it is made.** Closing the register is a snapshot, never a state. It is retained in full — including the two that failed — because the failures are the useful part, and because the next person to add an untraced claim to this document should be able to see what happened to the last eight.

**Result: 6 held, 2 were false — and the two failures were both load-bearing.**

| 39 | **RV-3028 `INT` can drive a GPIO directly** | ❌ **FALSE — "Interrupt Output; open-drain; active LOW; requires pull-up resistor" (datasheet pin table).** Pull-up added. **The app manual's advice to tie it to VBACKUP is rejected for this design**: an alarm asserting during a total outage would bleed the coin cell ~300µA into a pull-up serving a dead MCU. Ours goes to 3V3, which is 0V exactly when the coin cell is doing its job. **(Everything in this row transfers verbatim to the BM8563 that replaced the part — its pin table reads "Interrupt output (open-drain; active LOW)," verified in the batch-4 gates.)** |

**Batch 4's shape is new: 6 of 6 confirmed (recounted after rows 44 and 45 closed — the running tally had frozen at "4 of 5"), because the claims were checked *before* the decision instead of after.** The gates were written from the impact plan, the datasheet was read against them, and the swap executed only on passage — the first part selection in this project where the register preceded the commitment instead of auditing it.

**What made batch 3 different: every false claim was a *reasoned inference*, not a recalled spec.** Batch 1 failed on "I think the part does X." Batch 3 failed on "the part doesn't do X, **therefore** the design must be Y" — and the inference was never checked even though the premise was. #17 is the archetype: *TP4056 has no SYS rail* is true and cited; *therefore TP4056 is disqualified* is false and was never examined. **The register's standing rule caught the first kind and let the second kind straight through.**

**Amended standing rule:** *cite the datasheet line for the premise — and then state the inference the premise is being used to support, separately, so it can be attacked on its own.* An unchallenged "therefore" is as expensive as a fabricated spec, and it hides better, because the citation next to it looks like proof.

**Negative results need positive controls.** The first MP28167 sourcing check reported "NO RESULTS" for all three parts — including TPS63020, already confirmed at 10,374 in stock. **The control failed, so the method was discarded** (dead endpoints). Without it, a broken tool would have entered this register as a verified sourcing verdict. **Any negative claim here must name the control that proves the method works.**

**Method note that generalises beyond this project.** Batch 1's failures were claims about **capability** ("the charger exposes battery-absent", "the RTC has a TCXO"). **Batch 2's are mostly claims about *behaviour of a standard*** — what USB-C does, what a USB-A port does, what an LED needs. Those feel even safer than part capabilities, because they're "just how it works." **They are the same kind of claim and deserve the same suspicion**: #9 and #10 between them decide whether a documented safety warning is necessary or misleading.

**What this cost, and what it bought.** Each batch-1 row was a datasheet lookup, not a research project — all eight were closed in one sitting. Between them they forced one part change (BQ24074 → MCP73871), retracted one product claim (temperature compensation), caught one fatal BOM instruction (THERM), quantified two decisions that had been hand-waved (supercap backup, EEPROM wear), and surfaced one competitive regression nobody had noticed. **Every one of those would otherwise have been found by a board that didn't work.**

**The standing rule this leaves behind:** *if a decision in this document rests on a part doing something specific, cite the datasheet line. If no line can be cited, the claim is a guess and must be marked as one.* Two of the eight above were guesses wearing the same clothes as the other six, and nothing in the prose distinguished them until someone looked.

## Appendix: evidence log (why these decisions, condensed)

- **Witty Pi fails even on supported hardware**, not just mismatched OS: UUGear's own forum shows a stock-Raspbian user hitting "shutdown completes, power never cuts" because the firmware inferred "Pi is off" from a serial pin voltage that didn't behave as expected. A separate thread shows Witty Pi silently overwriting host timezone assumptions with UTC, corrupting schedules. Both are the same root issue — inferring host state instead of the host declaring it.
- **Independent convergence on the same architecture:** a Pi 5 owner, unprompted, built their own systemd/Python service reading a schedule and writing to the standard `/sys/class/rtc/rtc0/wakealarm` interface — for different reasons (Pi 5's RTC can only wake, not shut down) than the original motivating case.
- **Cross-platform, beyond Pi entirely:** Olimex's own MOD-RTC product (UEXT-connected RTC) has a direct, unanswered user request in its own comments for scheduled-wake capability. A completely unrelated platform (STM32MP1, on ST's own official forum) shows engineers needing bootloader-level (TF-A) patches just to get reliable wake-from-shutdown working.
- **The "host explicitly signals done" pattern generalizes**, independent of Pi: a Critical Link MityDSP (ARM9, NAND-based) forum thread shows the same architecture solved via a standard Linux technique — hook a power-cut script into the shutdown sequence itself (e.g. `/etc/rc0.d/S80KillPower`), after filesystems are already unmounted, so "safe to cut" falls out naturally instead of being inferred.
- **Immediate cut-on-command is safe, not just theoretically fine:** an experienced embedded engineer reports 100+ deployed systems, 10+ years, zero filesystem corruption, using exactly this pattern (host syncs/remounts read-only, then power is cut with no extra delay). For extra margin, Technologic Systems' TS-SILO product shows a precedented hold-up-capacitor approach as a real, shipping alternative if wanted later.
- **Bare-metal MCU hosts are simpler to integrate, not harder** than originally assumed — Linux needs a real (if standard) shutdown sequence; bare-metal firmware already knows unambiguously when it's done, no OS-layer indirection needed. Proven by the "powerTimer" project (see Scope section).
